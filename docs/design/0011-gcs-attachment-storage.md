# ochakai 設計ドキュメント 0011: 添付画像バイト列の GCS 移行

Status: Superseded in part by 0013(bytea との二重化は廃止され GCS のみに。
§2.1 の「未設定なら従来どおり bytea」と §2.4 の二重読み出し・再インライン
化はもう当てはまらない)
Date: 2026-07-18

## 1. 目的

設計ドキュメント 0008 §3.1 は添付画像のバイト列を PostgreSQL の
コンテンツアドレス blob 行に置き、「限界が来たら blob テーブルだけを
外部ストレージへ逃がす移行路が sha256 キーのおかげで閉じている」と
書いた。その限界が来た: ホスト環境のデータベース容量が画像で埋まり
始めた。本ドキュメントはその移行路を実際に開通させる。

## 2. 決定

### 2.1 バイト列だけが動く。モデルは動かない

blob は今までどおり SHA-256 でアドレスされる不変のコンテンツであり、
`attachment` 行(エントリ + ファイル名 → sha256)、リビジョン、
メタデータ(media_type, size)はすべて PostgreSQL に残る。動くのは
`blob.bytes` 列の中身だけ:

- `OCHAKAI_GCS_BUCKET` を設定すると、バイト列は GCS オブジェクト
  `blob/<sha256>` に住む。`blob` 行は残る(bytes は NULL)。
- 未設定なら従来どおり bytea。**「Docker だけでローカル起動」は壊さない**
  — compose 環境・統合テストは今までどおり GCS なしで完結する。

### 2.2 認証は ADC、secret-zero は維持

GCS クライアントは Application Default Credentials で認証する。
Vertex 埋め込み(設計ドキュメント 0001 §4)と同じパターンであり、
発行・ローテーションすべきトークンは増えない。Cloud Run のサービス
アカウントにバケット単位で権限を与える(デプロイガイド参照)。

### 2.3 書き込みは create-only、blob は消さない

- Put は `DoesNotExist` 前提条件付きの create-only。既存オブジェクトへの
  上書きは起きず、412(すでに存在)は成功として扱う — コンテンツ
  アドレスなので同じ名前は同じ中身である。
- GCS 書き込みはトランザクションの外(前)で行う。書き込み成功後に
  DB 側が失敗しても、残るのは参照されないコンテンツアドレス
  オブジェクトだけで無害(次の同一画像 attach が再利用する)。
- blob は今までどおり削除しない。リビジョンがハッシュで歴史上の内容を
  指せる性質は、置き場が GCS になっても変わらない。

### 2.4 移行は起動時バックフィル、読み出しは二重化

- 起動時、`OCHAKAI_GCS_BUCKET` が設定されていれば
  `bytes IS NOT NULL` の blob 行を GCS へコピーし、成功した行の bytes を
  NULL にする。sha256 単位で冪等なので、途中で落ちても次の起動が続きを
  やる。スキーマ移行(migration 0007: bytes の NOT NULL 制約を外す)と
  同じく、追加コマンドや手順は要らない — デプロイして env を足すだけ。
- 読み出しは bytes が NULL なら GCS、あれば bytea の二重化。バック
  フィルがどこまで進んでいても正しく読める。
- **ロールバックの注意**: バックフィルは追加的でない(bytes を NULL に
  する)ため、GCS 有効化後に GCS を知らない旧バイナリへ戻すと、移行済み
  blob は読めなくなる。デプロイのロールバック可能性が残っている間は
  env を足さないこと。デプロイガイドの version notes に明記する。

## 3. やらないこと

- **署名付き URL・リダイレクト配信。** 添付はこれまでどおり API が
  プロキシして返す(REST の ETag、MCP の image content block は不変)。
  クライアントから見て何も変わらないことが、この移行の合格条件である。
- **GCS からの blob GC。** 参照されない blob の回収は bytea 時代にも
  やっていない。ストレージ単価が桁で下がる以上、優先度はさらに低い。
- **S3 等の非 GCP バックエンド。** 設計ドキュメント 0003 のとおり
  ochakai は Google Cloud 専用。インターフェイス(`blob.Store`)は
  最小に保つが、実装を増やすためではなく、テストのフェイクのためにある。
- **エントリ本体・リビジョンの外部化。** 動かすのは画像バイト列だけ。
  テキストは Postgres が最適な置き場であり続ける。
