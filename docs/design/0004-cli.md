# ochakai 設計ドキュメント 0004: リモート CLI

Status: Draft
Date: 2026-07-16

## 1. 目的と位置づけ

シェルを持つコーディングエージェント(Claude Code、ヘッドレスエージェント、CI)と
人間の運用者に向けて、REST API の薄いクライアントとしての CLI を追加する。

動機:

- コーディングエージェントは `gh` / `gcloud` のような CLI を最も自然に使う。
  ツールスキーマが常時コンテキストを消費せず(`--help` を必要時に読む)、
  パイプで合成でき(`ochakai search ... | jq`)、サブエージェント・スクリプト・CI
  からそのまま呼べる。
- 現状の Claude Code からの接続手順(`gcloud run services proxy` + MCP 登録)は
  常駐プロセスが必要で、ワンショットのスクリプト用途に向かない。CLI は
  ID トークンを自分で取得して直接 HTTPS で叩くため、プロキシが不要になる。

**MCP を置き換えるものではない。** シェルを持たないホスト型エージェント
(claude.ai コネクタ、Claude Desktop、Gemini Enterprise の managed agents 等)には
MCP だけが入口であり、ochakai の対外的な主インターフェースは MCP のまま。
本ドキュメントは「サービス層は一つ、口は三つ(MCP / REST / CLI)」の三つ目を定める。
CLI は既存 REST API(`api/openapi.yaml`)の純粋なクライアントで、
**サーバー側に新しい面を作らない**。

## 2. 決定

1. **同一バイナリ**。`cmd/ochakai` に client 系サブコマンドを追加する。
   別バイナリ(`okctl` 等)は作らない — 配布物が一つで済み(`go install` /
   リリースバイナリ / 既存イメージ)、リポジトリの単一バイナリ方針(0001 §4)と揃う。
   代償はサーバー/CLI 相互の未使用依存のリンク(実測 +12MB、19→31MB)で、
   Cloud Run でも `go install` でも運用上の実害はないと判断した(vault /
   consul / cockroach と同型)。ただし `internal/apiclient` は
   `internal/service` を import せず **OpenAPI 契約(api/openapi.yaml)を
   ミラーした型を自前で持つ**(ワイヤ互換はテストで固定)。CLI 専用
   バイナリへの分割が必要になった時に依存ツリーごと切り出せる状態を保つ。
   分割の転換点は、brew 等での非開発者向け配布、または CLI とサーバーの
   リリース頻度の乖離。
2. **フラットな動詞コマンド**。リソースが knowledge 一種類しかないため
   `ochakai knowledge search` のようなグループ化はせず `ochakai search` とする。
   0001 決定 6 の `動詞_対象` 形式は「複数 MCP サーバー間での曖昧さ回避」が理由で、
   CLI では `ochakai` プレフィックスが名前空間を担うため衝突しない。
3. 既存の `serve` / `import-ossie` / `export-okf` は **DB 直結の server admin
   コマンド**としてそのまま残す。`ochakai help` では「server admin」と「client」を
   分けて表示し、役割の混同を防ぐ。
4. **引数なしの `ochakai` はヘルプを表示する**(serve へのフォールバック廃止)。
   bare 実行するのはほぼ CLI 利用者であり、`OCHAKAI_DATABASE_URL is required`
   で落ちるのは最悪の第一印象になる。コンテナイメージは `CMD ["serve"]` を
   明示しているためデプロイへの影響はない。
5. **接続先はローカル設定で選択できる**(v0.4.x 追加)。`ochakai use <url>` が
   接続先を `$XDG_CONFIG_HOME/ochakai/config.json`(既定
   `~/.config/ochakai/config.json`、Windows は `%AppData%\ochakai` —
   gh / gcloud と同じ解決順)に保存し、`ochakai use <name>` で切り替える
   (`docker context use` / `kubectl config use-context` と同型)。解決順は
   **`--url` > `OCHAKAI_URL` > config** — 明示が永続に必ず勝つ順序にすることで、
   人間の config に引きずられて CI・エージェントが意図しないインスタンスを叩く
   「隠れ状態」事故を防ぐ。`ochakai whoami` が「どこに・誰として・届くか」
   (接続先とその由来、提示する identity のローカル判定、`GET /health` での到達性)を
   表示する(`npm whoami` / `gh auth status` と同型)。当初 §7 で設定ファイルを
   作らないとしていたが、環境変数のみの UX は人間の運用者にとっつきにくい、
   という理由で覆した。config が持つのは URL のみ — トークンは引き続き存在しない。
6. **独自の OAuth ログイン(`ochakai login`)は作らない**。ユーザー資格情報の
   ADC は audience 指定の ID トークンを発行できず(§4)、認可サーバーは Google
   なので、login を実装しても `gcloud auth login` のラッパーにしかならない。
   資格情報の状態の可視化は `ochakai whoami` が担う。

## 3. コマンド体系

client 系コマンドと REST の対応(1:1、CLI 独自のサーバー呼び出しはない):

| コマンド | REST | 備考 |
|---|---|---|
| `ochakai search [query] [--type t]... [--status s]... [--tag x]... [--limit n]` | `GET /api/v1/knowledge` | verified が上位に来るのはサーバー側の挙動 |
| `ochakai get <type>/<id>` | `GET /api/v1/knowledge/{type}/{id}` | 既定出力は OKF 形式の Markdown |
| `ochakai create [-f file]` | `POST /api/v1/knowledge` | `-f` 省略時は stdin |
| `ochakai update <type>/<id> [-f file]` | `PUT /api/v1/knowledge/{type}/{id}` | 全置換(リビジョン保存はサーバー側) |
| `ochakai delete <type>/<id>` | `DELETE /api/v1/knowledge/{type}/{id}` | soft-delete |
| `ochakai compile --metric m [--dimension d]... [--filter "f op v"]... [--grain field:grain] [--model m] [--dialect d] [--limit n]` | `POST /api/v1/compile` | SQL を stdout、notes を stderr |
| `ochakai export [dir]` | `GET /api/v1/export` | tar.gz を展開して書き出す。`-` で tar.gz を stdout へ |
| `ochakai use [name \| url] [--name n]` | —(ローカル) | 接続先を config に保存・切替。引数なしで一覧(決定 5) |
| `ochakai whoami [--json]` | `GET /health` | 接続先とその由来・identity・到達性を表示(決定 5) |

グローバル: `--url`(または環境変数 `OCHAKAI_URL`、または `ochakai use` の選択。
優先順位は `--url` > `OCHAKAI_URL` > config)、
`--json`(整形出力を抑止し API レスポンスをそのまま出す)。

使用例:

```console
$ ochakai search "粗利" --type metric --type term --status verified
$ ochakai get metric/revenue
$ ochakai get metric/revenue | sed 's/.../.../' | ochakai update metric/revenue
$ ochakai compile --metric revenue --dimension orders.region \
    --grain orders.created_at:month --filter "orders.status = shipped"
$ ochakai export ./knowledge-backup
```

## 4. 認証

0002/0003 のとおりトークンという概念は持たない。CLI は Google ID トークンを
自力で取得して `Authorization: Bearer` に載せる。解決順:

1. `OCHAKAI_URL` が `http://`(ローカル開発、サーバー側は
   `OCHAKAI_INSECURE_DEV=true`)→ トークンなし。
2. ADC がサービスアカウント(metadata server / SA キー / impersonation)→
   `google.golang.org/api/idtoken` で audience = サービス URL の ID トークンを取得。
   ヘッドレスエージェント・CI はこの経路(gcloud バイナリ不要)。
3. ADC がユーザー資格情報 → `gcloud auth print-identity-token` に
   フォールバック(ユーザー ADC は任意 audience の ID トークンを発行できないため。
   Cloud Run IAM はユーザー ID トークンを受理する)。

actor 解決は従来どおり全面的にサーバー側(0002): 人間の gcloud 資格情報なら
human:本人メール、SA なら agent:SA メール。CLI は provenance 情報を一切送らない。

## 5. 入出力と終了コード

- **`get` の既定出力は OKF 形式**(YAML frontmatter + Markdown 本文)。
  `export` バンドルの 1 エントリと同一表現で、人間にもエージェントにも読みやすい。
- **`create` / `update` の入力は JSON(Knowledge スキーマ)と OKF Markdown の両対応**
  (先頭 `---` で判別)。`get` → 編集 → `update` がそのまま往復する。
- `search` の既定出力は 1 件 1 行(score、uri、status、title、一行説明)。
  機械処理には `--json`。
- `compile` は SQL を stdout に裸で出す(そのままファイルや次のツールへ渡せる)。
  notes と verified_queries の提示は stderr。
- 終了コード: `0` 成功 / `1` エラー(接続・認証・404 等)/
  `2` compile の 422(対応サブセット外)。`2` は「リクエストは正しく処理され、
  意図的に拒否された」ことを表し、エージェントが理由(stderr)と提示された
  ゴールデンクエリを読んで再計画するためのシグナル。

## 6. Claude Code からの利用

- `examples/claude-code/` に CLAUDE.md スニペット(またはスキル)を置く:
  `OCHAKAI_URL` の設定と代表的な 4〜5 コマンド例、
  「学びは `ochakai create` で書き戻す(draft で作られ provenance が残る)」の一文。
- `--help` はエージェントが読む前提で書く: 各コマンドに用途一行 + 実例 2 つ、
  出力形式の説明。ここが実質的なツール定義になる。
- MCP 接続(proxy 方式)も引き続き有効。ドキュメント上は
  「ホスト型エージェント = MCP、シェルのある環境(Claude Code 含む)= CLI を推奨」と案内する。

## 7. 作らないもの

- MCP の deprecate・機能削減(主インターフェースのまま)
- 新しいサーバー面(CLI 専用エンドポイント、認可、トークン)
- 独自の OAuth ログイン(`ochakai login`)— 決定 6
- ローカルキャッシュ・オフラインモード
- インタラクティブ UI / TUI
- リモート import(REST に import エンドポイントがないため。必要になったら
  REST 側の追加とセットで別ドキュメント)

## 8. 実装スコープ(v0.4)

1. `internal/apiclient`: REST クライアント(domain 型を再利用、ID トークン解決を含む)。
   テストは `httptest` + ゴールデン。
2. `cmd/ochakai`: client 系サブコマンド追加(stdlib `flag`。フレームワークは入れない)。
   help の admin / client 分離。
3. 依存追加は `google.golang.org/api/idtoken` のみ(Cloud SQL コネクタ経由で
   既に間接依存にあるものを直接依存へ)。
4. `examples/claude-code/` スニペットと README の接続手順更新
   (proxy 方式と CLI 方式の併記)。
5. 配布: `go install github.com/na0fu3y/ochakai/cmd/ochakai@latest` を README に明記。
   リリースバイナリ(checksums + provenance、0001 §6 と同水準)は任意の後続。
6. (v0.4.x)`use` / `whoami` と config ファイル(決定 5・6)。apiclient に
   `Identity`(ID トークンの email をローカル判定、actor 解決の正はサーバー)と
   `Health`(`GET /health`、Cloud Run 越しなら IAM 受理の証明を兼ねる)を追加。
