# ochakai 設計ドキュメント 0009: OKF/Git 往復と provenance の所有権

Status: Proposed
Date: 2026-07-18

## 1. 目的

設計ドキュメント 0005 §3.3 の「サーバー所有キー」ルール(`created_by` /
`verified_*` はペイロードから受け取らず、provenance は認証から来る)の帰結として
生じる問い——**export → Git → import の往復で provenance は誰のものになるのか**——に
明示的に答え、世界観として記録する(Issue #42)。

検討した選択肢:

- **案 A**: サーバー管理のインポート経路に限り `--preserve-provenance` を許す
- **案 B**: 「Git はレビュー経路」ワークフローを規範化する(コード変更なし)
- **案 C**: 現状維持を明文化する: provenance はインスタンス固有であり、
  バンドルは知識のみを運ぶ

**決定: 案 C を世界観として採用し、案 B をその上の推奨ワークフローとして
規範化する。案 A は不採用。**

## 2. 前提の修正: 問題は起票時の想定より狭い

Issue #42 は「往復で全エントリの provenance がインポート実行者に付け替わる」を
前提としたが、実装を精査するとそうではない。

`ochakai import` は既存エントリに対して Update(全置換)を発行し、Update は
`created_by` / `created_at` / `verified_by` / `verified_at` を**旧エントリから
引き継ぐ**。verified → verified の遷移では検証者を再スタンプしない
(`internal/service` の `applyVerification`: 新たに verified に**なった**ときだけ
actor を記録する)。つまり:

- **同一インスタンスへの export → 編集 → import 往復では、既存エントリの
  provenance は一切失われない。** 日常の Git レビュー運用はこのケースである。
- provenance がインポート実行者になるのは **Create 経路だけ**。すなわち
  (a) 空のインスタンスへの移行(全件が新規)、(b) 他インスタンスへの共有、
  (c) バンドル内の新規エントリ。

したがって本ドキュメントが答えるべき実質は「(a) 移行と (b) 共有のとき、
provenance を運ぶべきか」に絞られる。

## 3. 決定

### 3.1 世界観(案 C): DB が真実の源、バンドルは知識を運ぶ

**provenance はインスタンス固有の観測記録であり、可搬な属性ではない。**
`created_by` / `verified_by` は「このインスタンスで誰が入れ、誰が検証したか」を
サーバーが観測した事実であって、エントリが自己申告する主張ではない。

- エクスポートされた frontmatter の `created_by` / `verified_by` / `verified_at` は
  **参考情報(歴史的記録)**である。インポートで読み戻さないのは仕様であり、
  0005 §3.3 の「サーバー所有キー」行を本ドキュメントが世界観として裏付ける。
- 「Your knowledge is yours」との緊張は次のように整理する: **持ち出せるのは
  知識であり、バンドル(と Git 履歴)が旧インスタンスの provenance の
  アーカイブになる。** 移行しても記録は Git に残り、失われるのではなく
  「live な記録」から「歴史的記録」に変わる。
- 新インスタンスで importer が `created_by` / `verified_by` になるのは、
  むしろ正確な記録である: そのインスタンスにおいて知識を入れ、verified の
  状態に置いたのは importer 本人だからである。

### 3.2 ワークフローの規範化(案 B): Git はレビュー経路、verifier は取り込んだ者

Git を知識のレビュー経路にする組織の標準ワークフローを次のとおり規範化する
(ガイド化は #43 の CI レシピと併せて行う):

1. `ochakai export` した bundle を Git リポジトリに置く
2. 知識の変更は PR で行い、レビューを経て merge する
3. merge 後に `ochakai import`(将来は `ochakai sync`)を実行する。
   実行者は merge した人間、または CI のサービスアカウント
4. このとき記録される actor は「Git のレビューゲートを通したのが誰(何)か」の
   正確な記録になる: 人間が実行すれば `human:本人`、Workload Identity
   Federation 経由の CI なら `agent:ci-sa` = 「CI パイプラインが取り込んだ」

既存エントリの verified_by は §2 のとおり保持されるため、このワークフローで
検証者が「付け替わる」のは、PR で内容ごと status を動かしたエントリと
新規エントリだけである。これはレビューゲートの意味論と一致する。

### 3.3 案 A の不採用理由

1. **前提コマンドが存在しない。** 案 A が想定した `import-okf`(DB 直結)は
   0007 で削除済みであり、復活は「DB 直結は `serve` のみ」の決定と正面衝突する。
2. **API 側に作ると 0002 と矛盾する。** ochakai には認可機構がない
   (到達できた者は読み書きできる)。`--preserve-provenance` を「管理者だけ」に
   限定する手段がなく、role を作らない限り**誰でも任意の provenance を
   自己申告できる**ことになる。0002 の「認可ではなく記録で担保する」モデルは
   provenance が改竄不能な観測記録であることに全面的に依存しており、
   ペイロードから受けた瞬間に信頼のアンカーが消える。
3. **救いたいユースケース(移行)は §3.4 の運用で足りる。**

### 3.4 インスタンス移行の推奨運用

セルフホストの引っ越し・テナント移行では:

- 移行元の最終 export を Git(または保管場所)に残す。frontmatter の
  provenance がそのまま移行前の記録のアーカイブになる
- import は**移行専用のサービスアカウント**で実行することを推奨する。
  `created_by: agent:migrator@...` は「移行で入った」と一目で読め、
  移行実行者個人が全エントリの作成者・検証者に見えることを避けられる
- verified を移行後も維持したいかは組織の判断に委ねる。維持するなら
  status ごと import すれば「移行ゲートを通した」記録として verifier =
  移行 SA になる。再検証を求めるなら status を draft に落として import する

## 4. やらないこと

- **provenance の署名付きエクスポート。** 信頼はインスタンス運用者に置く。
  インスタンス間で暗号学的に provenance を運ぶ需要が実在してから検討する
  (おそらく過剰であり続ける)。
- **`--preserve-provenance`(いかなる経路でも)。** §3.3 のとおり。
  role/認可を導入する日が来たら(0002 §4)そのときに再検討してよい。
- **OKF SPEC への先回り。** SPEC 側に provenance の標準キーが定義される動きは
  引き続きウォッチし(GoogleCloudPlatform/knowledge-catalog)、標準化されたら
  写像を再検討する。それまで ochakai の出力は参考情報のままとする。

## 5. 影響

- コード変更なし。現行実装は本ドキュメントの世界観を既に満たしている。
- #43(Git CI / sync)の競合検出は「DB が源、Git はレビュー経路」を前提に
  設計する: サーバー側が Git 外(MCP 経由の agent write-back)で更新されて
  いたら警告し、上書きは明示フラグに委ねる、が本世界観と整合する。
- 0005 §3.3「サーバー所有キー」行に本ドキュメントが世界観の根拠を与える
  (0005 の改訂はしない)。0005 §4 の「リネームで provenance は作り直しになる」
  注意書きは、リネームが新 ID = Create 経路である以上、§2 の整理とも整合して
  正しいままである。
