# ochakai 設計ドキュメント 0024: リンクは本文から導出する

Status: Accepted(2026-07-20)
Date: 2026-07-20

## 1. 目的

ochakai のリンクは今、**本文とは独立した構造化フィールド**である。
書き手は `links: [{rel: measures, target: metrics/revenue}]` を
create / update のペイロードに手で書き、本文の markdown リンクとは
無関係に保存される。これは 3 つの理由で誤りである。

1. **OKF はリンクを構造化しない。** SPEC §5(Cross-linking)はリンクを
   素の markdown リンク(`/` 始まりのバンドル相対、または `./` 相対)
   としてのみ定義し、関係の種類は「リンク自体ではなく周囲の散文が伝える」
   と明言している。リンク型を宣言する frontmatter キーも `[[...]]` 記法も
   SPEC には無い。0023 が型の語彙を OKF に一本化したのと同じ理由で、
   **ochakai がリンクに独自の構造を発明している**状態である。
2. **`rel` は実際には使われていない。** 全コードを走査しても、
   `Link.Rel` の値で分岐する箇所は **1 つも無い**。唯一の参照は
   `internal/okf/okf.go` の表示用フォーマットだけである。0018 の
   semantic model → table すら、実際は `attrs.model`
   (`internal/store/store.go` の `rewriteReferences`)で辿っていて
   `rel` を見ていない。**機械可読な型として導入されたが、機械が読んで
   いない。**
3. **本文と links が二重管理で、しかも同期しない。** 書き手が本文に
   `[売上](/metrics/revenue.md)` と書いてもリンクにはならず、
   `links` に書いてもそれは本文に現れない。さらに `Move` の
   `rewriteReferences` は `links` 列と `attrs.model` しか書き換えず、
   **本文中の markdown リンクは壊れたまま放置される**(現状のバグ)。

## 2. 世界観: 本文が唯一の出所

0017 は「パスが住所」とし、0022 は「ファイル名が名前」とした。本改訂は
その系列の 3 つ目である — **本文が関係である**。

`links` は書き手が編集する入力ではなく、**本文から導出される派生値**に
なる。DB の `links` 列は消えず、書き込みのたびに本文から再計算される
インデックスとして残る。読み取り側(backlinks、get_context の 1 ホップ
展開)は**一切変更を要さない**。

## 3. 決定

### 3.1 `links` は入力から受け取らない

create / update のペイロード、および frontmatter からリンクを受け取る
経路を廃止する。`links` は応答には出力され続ける(派生値として)が、
入力としては無視される。

- `internal/mcpserver/mcpserver.go` の `writeIn.Links` を削除
- REST は `domain.Knowledge` に unmarshal しているので `links` は
  読み取られはするが、直後に本文からの導出で上書きされる = 実質無視
- `internal/okf/parse.go` の `splitLinks`(末尾 `# Links` セクションの
  畳み戻し)を削除 — セクションはただの本文になり、その中の markdown
  リンクが通常どおり抽出される。往復は保たれる
- `internal/okf/okf.go` の `# Links` セクション生成を削除 — リンクは
  すでに本文にあるので、書き足すと二重になる

### 3.2 `rel` は `text` に降格する

```go
type Link struct {
    Target string `json:"target"`
    Text   string `json:"text,omitempty"` // 本文中のアンカーテキスト
}
```

`rel`(関係の型)ではなく `text`(アンカーテキスト)である。SPEC が
リンクテキストを関係名と定めていない以上、これを型として扱うことはでき
ない。一方で「何と呼ばれてリンクされているか」は backlinks 表示に
そのまま効くので、捨てずに表示ラベルとして持つ。

`Text` は**任意**である。裸の `ochakai://tables/orders` や
autolink `<ochakai://tables/orders>` にはアンカーテキストが無い。空の
場合の表示は `domain.DisplayTitle(target)` — target の最終セグメントを
名前とする、0022 のルールそのままである。

分岐しているコードが無いため、この改名の破壊コストは JSON キーと
エクスポート形式に限られる。

### 3.3 抽出の対象

本文中の次の形を抽出する。

| 形 | 例 | Text |
|---|---|---|
| markdown リンク(バンドル絶対) | `[売上](/metrics/revenue.md)` | あり |
| markdown リンク(相対) | `[売上](./revenue.md)` | あり |
| markdown リンク(正規 URI) | `[売上](ochakai://metrics/revenue)` | あり |
| autolink | `<ochakai://metrics/revenue>` | なし |
| 裸の URI | `ochakai://metrics/revenue` | なし |

- **`http(s)://` は抽出しない。** 自環境の公開 URL
  (`https://<環境名>/metrics/revenue`)をリンクとして解決する案は
  **見送る**(§4)。サーバは自分の公開 URL を知らず
  (`OCHAKAI_URL` はクライアント側の設定)、環境依存の解決はバンドルの
  可搬性(0009)を壊す。
- **`.md` 以外のファイル参照は添付**である(0008 / 0013)。抽出はこれに
  触らない。画像記法 `![alt](x.png)` も同様に対象外。
- 相対パスは**エントリ自身の id が属するディレクトリ**を基準に解決する。
  `metrics/revenue` の本文の `./gross.md` は `metrics/gross` になる。
- target は**素の id で保存する**(`ochakai://` 接頭辞は剥がす)。
  `ListLinkingTo` は両形を照会し続けるので、既存行との互換は保たれる。
- 完全に同一の `(target, text)` の組は重複排除する。同じ相手を違う語で
  指すのは散文として正当なので、target だけでの排除はしない。
- 自分自身へのリンクは落とす。

### 3.4 抽出は正規表現で行う(コードブロックは除外)

markdown パーサ(goldmark)を導入して AST から取る案も検討したが、
**依存を増やさず正規表現で行う**。ochakai のパーサ層は現状すべて
文字列・正規表現ベースであり、抽出対象が §3.3 の 5 形に限られる以上、
AST の一般性は要らない。

ただし**フェンスドコードブロック(``` / ~~~)とインラインコード
(`` ` ``)の中は除外する**。裸の URI を拾う以上、コード例に書かれた
`ochakai://...` がリンクになるのは明確な誤りである。行単位でフェンスの
開閉を追い、フェンス外の行からインラインコードのスパンを取り除いてから
リンクを拾う。

### 3.5 Move は本文を書き換える

`rewriteReferences` の対象に**本文**を加える。これは新機能ではなく、
現状のバグ修正である — 今は `links` 列だけが直り、著者が本文に書いた
リンクは壊れたまま残る。

- 参照元エントリの本文中で `oldID` に解決するリンクの target を
  `newID` に書き換える。`ochakai://` 形はその形のまま、それ以外は
  バンドル絶対形 `/newID.md` にする(相対形は解決先が変わるため、
  絶対形に正規化するのが唯一予測可能な扱いである)
- **移動するエントリ自身の本文も対象**である。`x/a` の `./b.md` は
  `x/b` を指すが、`y/a` に移ると `y/b` を指してしまう。移動前の id で
  解決してから絶対形に書き換える
- 書き換え後に `links` を再導出する

本文の書き換えはコードブロックを除外する(§3.4 と同じ判定)。

### 3.6 マイグレーション 0015

既存の `links` を**本文に書き戻してから**再導出する。書き戻さないと、
次にそのエントリが更新された瞬間にリンクが黙って消える。

- リンクを持つ live エントリの本文末尾に `# Links` セクションを追記する。
  各行は `- [<rel>](/<target>.md)` — **旧 `rel` がアンカーテキストに
  なる**。これは「text を rel として使う」という本改訂の読み替えを、
  既存データにそのまま適用したものである。`rel` が空の場合は target の
  最終セグメントを使う
- `links` 列を、その追記後の本文から導出される形
  (`{target, text}`、`ochakai://` は剥離)に置き換える
- リビジョンスナップショットは**触らない**。履歴は当時の姿を保つべきで
  あり、0013 が id を書き換えたのは id が別テーブルのキーだったためで
  ある。`links` はキーではない
- **再エンベディング不要。** `embeddingText` は id・title・description・
  tags・question・body から作られ、links を含まない。本文への追記は
  embedding に影響するが、追記されるのは既にリンクとして持っていた情報
  であり、内容の変化としては軽微である。次の更新で自然に追随する

### 3.7 表層の更新

- `internal/webui/index.html`: **リンク編集欄を削除**する。Links タブは
  読み取り専用の表示になり、`text` があればそれを、無ければ
  `DisplayTitle(target)` を表示する。「リンクは本文から」の一行を
  本文編集欄の近くに置く
- `api/openapi.yaml`: `Link` スキーマの `rel` を `text` に、
  write リクエストから `links` を外し、`readOnly` として記述する
- `internal/mcpserver/mcpserver.go`: `create_knowledge` /
  `update_knowledge` の説明から links を外し、body の説明に
  「他エントリへのリンクは本文に markdown で書く」を加える
- `README.md`: リンクの説明を「本文の markdown リンクがそのまま関係に
  なる」に改める

## 4. やらないこと

- **`https://<環境名>/...` の解決。** サーバに公開 URL の設定を足せば
  可能だが、環境依存の抽出はバンドルの可搬性を壊す。必要になったら
  別ドキュメントで扱う
- **`[[wikilink]]` 記法。** SPEC に無い。0001 が言及していたが、
  本改訂は SPEC の記法だけを正とする
- **リンク型(`rel`)の別形での存続。** 型付きの関係が本当に要ることが
  分かったら、SPEC §4.1 の拡張キーとして attrs に置くのが筋である。
  0018 の `attrs.model` が既にその形をしている
- **リンク解決の検証。** 存在しないエントリへのリンク(dangling)は
  許容する。backlinks は live エントリを JOIN するので実害が無く、後から
  そのエントリが作られれば自然に繋がる
- **`links` 列の廃止。** 派生値だが、backlinks の containment クエリ
  (`links @> $1`)が依存している。導出して同じ列に入れる限り、
  読み取り側は変更不要である

## 5. 壊れるもの

- **`links` を送っていたクライアント。** 無視される。本文に markdown
  リンクを書く形に移行が必要
- **`Link.rel` を読んでいたクライアント。** JSON キーが `text` になり、
  値の意味も「関係型」から「アンカーテキスト」に変わる
- **エクスポートされたバンドル。** `# Links` セクションは生成されなく
  なる(本文に既にあるため)。マイグレーション 0015 で書き戻された分は
  本文の一部として出力される
- **Web UI のリンク編集欄。** 消える
