# ochakai 設計ドキュメント 0026: Cloud Storage 一本化(各インスタンスが正本を投影する)

Status: Proposed(2026-07-24)
Date: 2026-07-24

## 1. 目的

ochakai は今、状態を **Cloud SQL PostgreSQL** に置く(0003)。本ドキュメントは
その代替として、**永続層を Google Cloud Storage(GCS)一本にする**構成を提案し、
実験でその成立性を検証する。

動機は 0003 と同じ「簡略化」の延長線上にある。0003 は認証を cloudrun-iam に
一本化し、静的トークンという概念ごと捨てた。本改訂はその姿勢を**永続層にも
適用する** — 常時稼働の VM(Cloud SQL)を持たず、シークレットも VPC も増やさず、
実行時依存を **{Cloud Run, GCS} の 2 つ**に畳む。

核となる設計判断は **「索引をどこにも焼かない」** ことである。正本は GCS
オブジェクト、索引は**各 Cloud Run インスタンスが起動時に正本を RAM へ投影した
もの**とし、以後は検索リクエストが差分を引き込む。この「builder を持たない
要求駆動」の形にすると、事前ビルド索引に付随して湧いてくるもの — 定期実行の
トリガ(Cloud Scheduler)、専用ビルドジョブ(Cloud Run Job)、ビルダーの相互排他
(GCS ロック)、torn build の鮮度窓 — が**まとめて消える**。

これは 0003 を supersede しない。0003 の核(GCP 前提・IAM 認証・シークレット
ゼロ・データ可搬性)は**むしろ強まる**。置き換わるのは「状態の置き場所が
Cloud SQL である」という機構だけである。

### 1.1 出発点となった観察

ochakai の DB が本当にやっている仕事を棚卸しすると、以下に尽きる。

- 数千〜数万件の JSON ドキュメント(エントリ)を、複数インスタンスから
  **原子的に**、**履歴つきで**永続化する
- その上で日本語のレキシカル検索とベクトル検索を提供する

そして検索は、コード自身が既に「ANN 索引は要らない、全走査で十分」と認めて
いる(`internal/store/migrate.go`)。数万件 × 768 次元 = 数十 MB は、プロセス内で
総当りして数ミリ秒で終わる規模である。**DB は検索エンジンである必要がなく、
「数十 MB の JSON を安全に永続化する」ことが本質**である。これを認めると、
Cloud SQL でなければならない理由が消える。

さらに小〜中規模(0001)では、**索引そのものがインスタンス内 RAM に丸ごと載る**
(§4)。索引が RAM に載るなら、それを事前に焼いて配る builder も要らない。各
インスタンスは正本を自分の RAM へ投影し、差分で追随すればよい。

## 2. 世界観: オブジェクトが正本、RAM 投影が索引

0017「パスが住所」、0022「ファイル名が名前」、0024「本文が関係」の系列に、
本改訂は 4 つ目を足す — **オブジェクトが状態である**。

```
                      ┌─────────────────────────────┐
   書き込み(CAS)  →  │  GCS: 1 エントリ = 1 オブジェクト │  ← 正本(source of truth)
                      │  entries/<id>.json (versioned)  │    バージョニング = 履歴
                      │  vectors/<id>.f32 (content 鍵)  │    埋め込みは書込時に確定
                      └───────────────┬─────────────────┘
                          LIST(強整合)│ 起動時: 全 GET / 定常: 差分 GET
                                      ▼
   読み取り        ←  各 Cloud Run インスタンスの in-RAM 索引(private・ephemeral)
                      ・trigram レキシカル ・float32 ベクトル ・逆リンク写像
```

- **正本**: エントリごとに 1 つの GCS オブジェクト。書き込みは
  `ifGenerationMatch` によるオブジェクト単位の CAS。オブジェクト
  バージョニングが `knowledge_revision` を置き換える。
- **索引**: **どこにも焼かない。** 各インスタンスが RAM に持つ、正本の投影。
  不変の共有成果物ではなく、プロセスの寿命だけ生きる私的な派生キャッシュ。
- **配信**: 各インスタンスは起動時に LIST + GET で索引を組み、以後は LIST で
  差分を検知して変更分だけ GET して更新する。すべて**リクエスト処理系の内側**で
  起きる。builder も scheduler も job も存在しない。

## 3. 検証: プロセス内検索は pg_trgm を再現できるか

**前提**: 検索を DB から出して各インスタンスのプロセス内に移すには、
`internal/store/store.go` の `SearchLexical` が使う `pg_trgm.similarity()` の
ランキングを、外で忠実に再現できねばならない。

**方法**: ochakai 自身の設計ドキュメント 24 本を見出しで分割した 208 エントリ、
本文由来の 142 クエリで、`SearchLexical` の採点式(similarity + ILIKE の +0.3 +
verified の +0.05)を全 29,536 ペアで両実装採点し突き合わせた。

**結果**:

| 指標 | 値 |
|---|---|
| top-10 順位の完全一致 | 141/142(99.3%) |
| top-10 集合の一致 | 141/142(99.3%) |
| 平均 Jaccard@10 | 0.9987 |
| スコア最大差 \|Δ\| | 0.0027(10位の同点入れ替わり 1 件のみ) |

唯一の差の原因も特定した。pg_trgm はマルチバイトのトライグラムを **CRC32 で
3 バイトに圧縮**して格納する(208 エントリ中 5 件で衝突)。差は常に一方向で、
**プロセス内実装のほうが真のトライグラム集合に忠実、PostgreSQL 側が近似**。
品質は落ちるどころか、わずかに上がる方向である。

**この構成の要点**: 本構成の配信層はこの**プロセス内 trigram スコアラそのもの**で
あって、SQLite FTS5 の bm25 のような別ランキングを挟まない。したがって
**99.3% の忠実性はプロキシではなく配信パスの実測値**であり、「事前ビルド索引に
別エンジンを使うと検索品質が変わる」という不確実性が構造的に存在しない。

## 4. 検証: 索引は RAM に載るか

プロセス内索引を各インスタンスの RAM に置く。10 万件までの実測で成立性を見る。

| エントリ | 本文 | レキシカル | ベクトル | 合計/インスタンス | 起動時構築 |
|---|---|---|---|---|---|
| 5,000 | 8.5 MB | 2.7 MB | 15.5 MB | 26.6 MB | 0.5s |
| 20,000 | 34 MB | 10.7 MB | 61.9 MB | 106.6 MB | 2.1s |
| 100,000 | 170 MB | 53.7 MB | 309.6 MB | 533.3 MB | 10.8s |

- **〜2 万件**: 合計 107 MB / 2.1 秒。512 MB〜1 GB のインスタンスに素直に載り、
  起動プローブの範囲に収まる。**本構成の快適域**。
- **10 万件**: 533 MB × インスタンス数 + 10.8 秒構築で、RAM とコールドスタートの
  両方が壁になる。**本構成の却下域**。ここは §14 の「事前ビルド共有スナップショット」
  か AlloyDB に譲る。

### 4.1 ベクトルは全走査

ベクトルはフラットな float32 配列として RAM に置き、Go でストリーム走査する。
`migrate.go` が「数千行なら全走査で十分」と認めている全走査を、そのまま各
インスタンスに移すだけなので結果はビット単位で同一。

| エントリ | 全走査/クエリ | ヒープ |
|---|---|---|
| 5,000 | 5 ms | 15 MB |
| 20,000 | 19 ms | 62 MB |
| 100,000 | 93 ms | 294 MB(却下域) |

10 万件前後で 93 ms に達し ANN 索引が要るが、これは pgvector 側も同条件なので
**永続層選択とは独立**の課題。

## 5. 索引: 正本の in-RAM 投影

### 5.1 コールドスタート(全投影)

インスタンス起動時に一度だけ:

1. `entries/` を **LIST**(強整合、`generation` を GET せず取得)。
2. 全エントリを GET(または §14 の任意 `pack` オブジェクトで 1 GET)。
3. RAM に 3 構造を組む — trigram レキシカル索引、float32 ベクトル配列、
   本文由来(0024)の逆リンク写像。
4. 各エントリの `generation` を「投影済み世代」として保持する。

コストは §4 実測どおり 5k=0.5s / 20k=2.1s。この間 Cloud Run は起動プローブに
CPU を割り当てるので同期構築で問題ない。

### 5.2 定常運転(差分投影・要求駆動)

検索リクエストを処理する際、インスタンスは**スロットル付き**で鮮度を追随する:

- 最後の LIST から一定の**活動時間**が経っていれば(壁時計トリガではなく、
  検索が来たときにのみ判定)、`entries/` を再 LIST する。
- 返る `generation` を投影済み世代と突き合わせ、**新規/更新/削除**を判定。
- 変更のあった id だけを GET し、trigram・ベクトル・逆リンクを**増分更新**する。

書き込みは 200/日(§10)= 約 7 分に 1 件。ある検索が LIST を引いたときに変わって
いるエントリは 0〜数件で、差分更新はミリ秒で終わる。**一括ビルドは存在しない。**
重い処理はコールドスタートの全投影だけで、それは起動時同期に閉じている。

## 6. 書き込みプロトコルと網羅性

### 6.1 GCS の意味論(前提)

以下の GCS の**公表された**性質だけに依存する
(cloud.google.com/storage/docs/consistency)。

- オブジェクト一覧(LIST)は強整合、read-after-write も強整合
- `ifGenerationMatch` がオブジェクト単位の compare-and-swap を与える
- `objects.list` は各オブジェクトの `generation` を返す(GET せず差分判定できる)
- オブジェクトの `timeCreated`/`updated` は**サーバ生成の時刻**(自前時計を
  信用せずリース期限等を判定できる。本構成では現状不要だが将来の余地)

### 6.2 書き込み: 3 つのルール

**ルール 1 — 書き込みは CAS。** `Update` は GET(gen N)→ 変更 →
PUT `ifGenerationMatch=N`。競合すれば 412 で弾かれ再読込してリトライ。これは
**現状の PostgreSQL より強い**。現 `Update` は `WHERE id=$1` の無条件 UPDATE で
同時編集が黙って last-write-wins する(§10-1)。GCS 移行は競合検出を獲得する。

**ルール 2 — 指される側を先に書く。** リビジョンオブジェクトを、それを指す
エントリより先に create-only PUT。どのクラッシュ地点でも孤児リビジョン(GC 可能)は
残りうるが**宙吊り参照は生じない**。トランザクションなしで entry + revision の
2 オブジェクト原子性を得る。

**ルール 3 — 埋め込みは書込時に確定する。** エントリを作成/更新したインスタンスが
その場で Vertex を呼んで `vectors/<id>.f32` を書く。鍵は埋め込み対象テキストの
内容ハッシュで、**テキストが不変なら再計算しない**。索引の差分更新は保存済み
ベクトルを GET するだけで、**Vertex を二度と呼ばない**(埋め込みコストは
コーパスサイズではなく変更率に比例する)。

### 6.3 網羅性(completeness)

正本はオブジェクト、LIST は強整合。任意のインスタンスが LIST すれば committed な
全エントリが見える。したがって **「ack した書き込みは、各インスタンスがスロットル窓
内で再 LIST した時点で必ず索引に入る」**。焼くビルドが無いので **torn build も無く**、
永久取りこぼしは構造的に起きない。これは事前ビルド索引方式(§14)がランダム並行
スケジュールのモデル検査を要したのに対し、本構成では**「LIST は強整合・refresh は
再 LIST する」の一行**で従う。単純さは本構成の核心的な利点である。

## 7. 調整: 共有ロックが要らない

事前ビルド方式のロックは「単一 builder の相互排他」のためにあった。本構成には
**守るべき共有可変成果物が無い**。各インスタンスは不変の正本オブジェクトから
私的な投影を作るだけで、互いに干渉しない。

| 共有書き込み対象 | 競合 | 手当て |
|---|---|---|
| entries/*(正本) | オブジェクト単位 | `ifGenerationMatch` CAS で分離済み |
| vectors/*(埋め込み) | オブジェクト単位 | create-only、内容鍵で冪等 |
| usage/<instance>/*(§9) | 構造的に無し | インスタンスごとに別オブジェクト |

**グローバルロックもリーダー選出もゼロ。** これが「依存を減らす」を運用面でも
実現する要。

## 8. CPU-after-response 問題が消える

Cloud Run はレスポンス送出後に CPU をほぼ 0 に絞る。事前ビルド方式はこれを
「重い一括ビルドをどこで走らせるか」という難題として抱え、Cloud Run Job か
min-instances=1(= アイドル $0 が死ぬ)を強いられる。

本構成では**重い処理がコールドスタート同期(CPU 割当あり)に閉じ、定常の差分
更新はリクエストパス上のミリ秒処理**なので、レスポンス後のバックグラウンド重
処理がそもそも発生しない。**min-instances も Job も Scheduler も不要**で、
scale-to-zero のアイドル $0 と両立する。

## 9. usage カウンタ

- 各インスタンスが `usage/<instance-id>/<yyyymmdd>.json` に**自分の分だけ**書く
  (インスタンス別なので 1 オブジェクト 1write/秒制限にもロック競合にも当たらない)。
- 読み取りパスから外し、インスタンス内バッファ + 定期フラッシュ(走っている
  インスタンス自身が行う。ジョブ不要)。合計が要るときはシャードを GET して足す。
- at-least-once、クラッシュ時ロス許容 — 現 `internal/store/usage.go` は既に
  「usage recording must never fail a read」と書いて失敗を握り潰している(§10-2)。
  新たな妥協ではない。

1000 人規模で人気エントリのカウンタが GCS の書込レートを超える問題は、本構成の
規模上限(§12)では**登場しない**。超えるなら Firestore の分散カウンタ /
BigQuery 追記に逃がす(その規模はそもそも §12 で Cloud SQL/AlloyDB 域)。

## 10. コスト

`asia-northeast1`・Standard、5,000 エントリ、書き込み 200/日、読み 2,000/日。
公表標準レートに基づく計算(実請求では未検証)。

| 項目 | 月額 | 備考 |
|---|---|---|
| 読み取り(検索・ブラウズ) | ~$0 | RAM 索引が答える。GCS 操作が出ない |
| 鮮度追随の LIST(Class A) | 数百円〜 | **スロットル依存**。要調整(下記) |
| 書き込み(CAS PUT + ベクトル) | $0.005 | Vertex 埋め込み費は別 |
| 保存(20 世代バージョニング込み) | $0.005 | 無視できる |
| **アイドル時** | **$0** | scale to zero、常時稼働ゼロ |
| (参考)現行 Cloud SQL f1-micro | $9.4 | 常時稼働 |

**落とし穴は鮮度追随の LIST。** `objects.list` は Class A(Class B の 12.5 倍)。
無節操に LIST すると Cloud SQL を超えうる。手当ては 2 つ、いずれかで足りる:

1. **スロットルを活動 60 秒に 1 回**(かつ検索トラフィックがある間だけ)、
   インスタンス内で 1 回の LIST を並行検索で共有する。
2. **任意の `manifest.json`(id→generation)を書込時に CAS 更新**し、追随を
   その 1 GET(Class B)で行う。200 writes/日ならマニフェストは hot にならず
   競合も出ない。ただし PUT 後・マニフェスト更新前のクラッシュで新規エントリが
   次の全 LIST 再照合まで検索に出ない(fetch by id は強整合で引ける)。→
   マニフェストは hint、**全 LIST を truth** とし、コールドスタートで必ず全 LIST
   再照合して自己修復する。

**コスト単独では移行の動機にならない**($9.4 は工数に見合わない)。動機は
「アイドル $0・依存を 2 つに・可搬性が runtime の事実」であって、コストは副産物。

## 11. 鮮度モデル

- **fetch by id・ブラウズ**: GCS を直接読むので**強整合**(read-your-writes)。
  「保存 → 詳細を開く」は正しく動く。
- **検索**: そのインスタンスの最後の LIST 時点の鮮度。上限はスロットル窓(例 ≤60
  秒)。温まったインスタンス上では次の追随まで遅延、**冷えたインスタンス上では
  常に最新**(現 LIST から組むため)。
- **インスタンス間の鮮度スキュー(事前ビルド方式に無い癖)**: 追随タイミングが
  ずれるため、同じ検索でも捌くインスタンスによって数十秒ぶんだけ結果が違いうる。
  共有スナップショットは全インスタンスが同一世代を載せるのでこの癖が無い。
  **手当て**: レスポンスに投影世代の鮮度を出す/wiki 級として許容。

「検索索引が数十秒遅れる wiki」として製品説明可能。運用必須条件として
**投影世代の鮮度をメトリクスに出す**(静かに古くなるのを検知する)。

## 12. スケール上限(正直な棚卸し)

限界は「人数」ではなく「**エントリ数**」で決まる。本質的コストは**索引が RAM に
載ること**と**コールドスタートで全投影すること**。

| 軸 | 5,000 | 20,000 | 100,000 |
|---|---|---|---|
| RAM/インスタンス | 27 MB | 107 MB | 533 MB(却下) |
| コールドスタート全投影 | 0.5s | 2.1s | 10.8s(却下) |
| ベクトル全走査/クエリ | 5 ms | 19 ms | 93 ms |

- **〜2 万件が本構成の快適域。** それを超えると RAM × インスタンス数と
  コールドスタート時間が壁になり、§14 の「事前ビルド共有スナップショット」方式
  (RAM 2.4 MB・1 GET)へ移るべき。全社カタログ級は Cloud SQL / AlloyDB。
- 1000 人規模では usage が最初に壊れる(§9)。これも永続層と独立。

この設計は 0001「規模より信頼密度」「小〜中規模チーム」という自己規定に最も
素直に一致する。**全社 1000 人カタログに売るなら Cloud SQL / AlloyDB を維持
すべき**で、その規模で GCS 一本を成立させる作業は ochakai 本体より大きい分散
システムの構築になる。

## 13. 前提条件(どの道を選んでも必要な修正)

以下は移行の前提であると同時に、**現状のままでも直すべき**バグ/負債である
(コードで実在を確認済み)。Cloud SQL 上で先に入れれば移行の切り分けになる。

1. **`Update` に楽観ロックがない**(`internal/store/store.go` の `Update` は
   `WHERE id=$1 AND deleted_at IS NULL` の無条件 UPDATE)。人間とエージェントの
   共同編集が黙って last-write-wins する。GCS の `ifGenerationMatch` と同じ
   意味論(期待 generation / updated_at の事前条件)を先に PostgreSQL 上で導入。
2. **usage の同期書き込み**。`RecordEvents` を読み取りパスから外し、インスタンス内
   バッファ + 定期フラッシュにする。どのバックエンドでも必要。
3. **ILIKE のワイルドカード未エスケープ**。`SearchLexical` の substring boost が
   `%`/`_` を含むクエリで全エントリに +0.3 を付け順位を平坦化する。プロセス内
   実装に移せば自然に消えるが、現状でもバグ。

## 14. 検討した代替(否定せず並べた上での棄却)

- **事前ビルドの共有 SQLite スナップショット + builder**: 索引を「起動時に組む」
  から「あらかじめ焼いてある」に変え、各インスタンスが 1 GET で ephemeral disk に
  載せる(10 万件 831 MB をヒープ 2.4 MB で引く、pure-Go modernc SQLite で FTS5
  trigram + bm25 を実測確認済み)。**10 万件級ではこちらが正しい** — RAM を
  2.4 MB に抑えコールドスタートを 1 GET にする。代償が builder / scheduler or
  job / lock / torn build の鮮度窓、そして配信層が bm25 になることで §3 の
  忠実性がプロキシに落ちる点。**〜2 万件では本構成(RAM 投影)が単純さで勝ち、
  それ以上はこの方式が RAM とコールドスタートで勝つ**。規模で住み分ける。
- **`pack` オブジェクト(上限規模では実質必須、§16 実測で格上げ)**: 全エントリを
  畳んだ 1 オブジェクトを置き、コールドスタートを N GET から 1 GET に落とす。
  §16 の実測で「2 万件の個別 GET は WAN 17 秒・in-region でも 2 万往復」と判明した
  ため、〜2 万件域では pack を既定にすべき(数千件までは個別 GET でも実用)。
  **pack を焼くのは専用ジョブではなくコールドスタートしたインスタンスが CAS で
  書き戻す**ことに限れば builder-less を保てる — pack は不変・世代付きの派生物で、
  古くても正本 LIST との差分で各インスタンスが埋める。
- **Cloud SQL 継続(現状)**: 全要求を満たす唯一の既製品。不満はアイドル $0 に
  できない一点。**コストが動機なら移行しない**が正解。接続数は f1-micro で 6〜8
  インスタンスで枯渇し 100 人前に upsize が要る(GCS は接続上限がない)。
- **AlloyDB / AlloyDB Omni**: PG 互換で移行は楽だが最小構成でも月 $100 超。数万件に
  ScaNN と列指向は過剰。全社カタログ級になった時の逃げ道として保持。
- **Cloud Spanner**: 技術的には唯一の全部入り(日本語 FTS + ベクトル + 強整合 +
  運用ゼロ)。フリーティアなし・月 $22〜。**検索品質が動機なら**唯一の候補だが
  コスト動機とは真逆。
- **Firestore**: GCS 一本とほぼ同じ崖(コールドスタートで N ドキュメント読むか)と
  同じ回避策(投影/スナップショット)を持つ。分散カウンタで §9 が解け接続上限も
  ない点は GCS より楽。**依存を最小化する**一点で GCS を採る。
- **DuckDB(配信層)**: FTS が Porter ステマー(英語)ベースで **trigram を
  持たず日本語部分一致を素で満たせない**(duckdb discussions #16071)。加えて
  cgo + 約 30 MB のネイティブライブラリで `CGO_ENABLED=0` にできず、pure-Go の
  現コードベース(pgx)の姿勢に合わない。**却下**。ビルド/分析の任意オプション
  (GCS 上 Parquet 直読み等)としてのみ将来余地。

## 15. 壊れるもの / 移行

- **`internal/store` の書き換え**。pgx 依存の `Store` が GCS クライアント +
  in-RAM 索引(ローダ + 差分追随器)に替わる。`Migrate`(versioned SQL)は不要。
  **builder / job / scheduler は作らない** — それが本設計の主眼。
- **埋め込みを書込パスへ**。Vertex 呼び出しを保存時に行い `vectors/<id>.f32` を
  内容ハッシュ鍵で書く。
- **既存データの移行**。Cloud SQL の各行を `entries/<id>.json`、履歴を
  オブジェクトバージョンか `revisions/<id>/<rev>.json` として書き出す一括
  エクスポート。
- **ローカル開発**。docker compose の PostgreSQL が不要になり、GCS エミュレータ
  (fake-gcs-server)かローカルファイルシステムのアダプタに替わる。
- **0003 の環境変数**。`OCHAKAI_DATABASE_URL` / `OCHAKAI_DB_IAM_AUTH` が
  `OCHAKAI_GCS_BUCKET`(添付で既出)系に置き換わる。

破壊的変更が大きいため、リリースは独立した専用の列車に載せる。段階移行として
「Cloud SQL を正本のまま、各インスタンスが GCS 投影で検索だけ担う」中間形も
取りうる(§13 を入れた後)。この中間形は §17 の残リスク(in-region 投影レイテンシ・
鮮度スキュー・検索品質)を全面書き換えなしに実地検証する足場になる。

## 16. 実測: 実 GCS 検証(ochakai-example, 2026-07-24)

§6.1 の GCS 意味論と §5/§10 のコスト・投影モデルを、実バケット
(`asia-northeast1`・Standard・versioning 有効・2 万件・約 1.8KB/オブジェクト)で
検証した(ハーネス `cmd/gcsverify`、手元 WAN 経由。前提バグ #106/#107/#108 修正後)。

| 検証項目 | 対応 | 結果 |
|---|---|---|
| ifGenerationMatch CAS の 412 | §6.2 ルール1 | ✅ stale 世代の書き込みが 412 で弾かれる(勝者総取り) |
| create-only(DoesNotExist)の 412 | §6.2 ルール2 | ✅ 既存で 412 / 不在で成功(revision-first が安全) |
| read-after-write 強整合 | §6.1 | ✅ 20/20 の即時読みが直前書き込み値を返す |
| objects.list が Generation を返す | §5.2 | ✅ GET せず世代取得(差分検知が成立) |
| バージョニングで旧世代保持 | §6.2 | ✅ 旧 generation が GET 可(履歴が knowledge_revision を置換) |
| 単一オブジェクトの mutation ~1/s 制限 | §9/§10 | ✅ 同一オブジェクト連打で **429** を実観測(usage をオブジェクト化しない根拠を裏取り) |
| LIST ページング | §10 | ✅ 2 万件 = **21 ページ = 21 Class-A ops** / 1.2 秒。差分検知の LIST は激安 |
| 投影ロード(LIST + 2 万並列 GET) | §5.1 | ⚠️ **17 秒**(GET p50=48ms/WAN, c=64, 1267 obj/s)。下記参照 |

**設計に効いた発見 — 素朴な N 個別 GET のコールドスタートは重い。** 2 万件を
個別 GET すると WAN で 17 秒(GET は往復レイテンシ律速、p50 48ms)。同一リージョン
(Cloud Run→GCS)なら 1 桁速くなり数秒に落ちるが、それでも 2 万往復は本質的に重い。
これにより §14 の **`pack` オブジェクト(1 GET で全件)を「任意の最適化」から
「上限規模での実質必須」に格上げする**。36MB の pack を 1 GET する方が、in-region
でも 2 万個別 GET より桁で速い。**LIST は逆に激安(21 Class-A)** なので、差分検知
(§5.2)と鮮度追随のコストモデル(§10)は追認された。

実費は約 32 円(2 回のシード分の Class-A PUT が主。100 円ゲート内)。

## 17. 未解決 / リスク(残)

- **投影ロードの in-region 実測**: §16 は WAN 実測。Cloud Run(asia-northeast1)から
  同一リージョン GCS への per-GET レイテンシと、`pack` 1 GET 方式の実転送速度・
  コールドスタート全体時間は、実デプロイでの再測が要る。
- **鮮度追随の LIST 頻度**: §10 のスロットル設計(活動 60 秒に 1 回/マニフェスト
  切替閾値)は実トラフィックでの検証が要る。LIST 単価が安いことは §16 で確認済み。
- **インスタンス間の鮮度スキュー**(§11): 同一クエリの結果がインスタンスで
  ぶれる。UX 上どこまで許容されるかは未評価。
- **FTS5(bm25)/2 文字クエリ**: trigram は 3 文字未満を引けない。日本語 2 文字
  (添付・検索・認証)は最頻出。in-RAM なら LIKE 全走査フォールバックが素直
  (索引不要、5k で数 ms)で、事前ビルド SQLite 方式より本構成のほうがこの縮退が
  軽い。検索品質そのもの(§3 スコアラの実クエリ体感)は次の実験対象。
