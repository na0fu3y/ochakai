# ochakai 設計ドキュメント 0026: Cloud SQL 全廃 — Cloud Storage 単体で提供できるか

Status: Proposed(2026-07-23 調査・検証)
Date: 2026-07-23

## 1. 問い

「Cloud SQL を全廃し、Cloud Storage だけで ochakai を提供できるか」を
調査・検証する。動機はコストと運用の簡略化である。デプロイ手順書
([deploy/cloudrun/README.md](../../deploy/cloudrun/README.md))の見積り
どおり、月額の支配項は Cloud SQL(`db-f1-micro` で ~$9–10)であり、
Cloud Run はゼロスケールでアイドル時 ~$0、Vertex 埋め込みは任意。
**Cloud SQL を落とせれば、月額はストレージ + リクエストのセント単位まで
下がる。** これは「$10/month」を掲げる本プロジェクトの価値観に合致する。

結論を先に言う。**技術的には可能で、ochakai の「小ささ」(README:
trust density over volume)と OKF バンドルを正本とする思想に、むしろ
自然に嵌まる。ただし単一の重大な譲歩 — 書き込みインスタンスを 1 つに
固定する — を受け入れる必要がある。** 以下、現状の棚卸し、置換設計、
実コードによる検証、そして「無理が出るところ」を順に述べる。

## 2. 現状: PostgreSQL が担っている役割

`internal/store` を棚卸しすると、Cloud SQL は次の 7 つを提供している。

| # | 機能 | 実装 |
|---|---|---|
| 1 | 主キー参照(id で 1 件取得) | `SELECT … WHERE id=$1` |
| 2 | 語彙検索 | `pg_trgm` の trigram 類似度 + `ILIKE` 部分一致フロア + verified ブースト(`SearchLexical`) |
| 3 | 意味検索 | `pgvector` の コサイン距離(`SearchVector`、任意) |
| 4 | 関係クエリ | JSONB 包含 `links @> …`(バックリンク)、`attrs->'spec'->'metrics' @> …`(モデル解決) |
| 5 | マルチ行トランザクション | `Move` の参照書き換え、`Create`+リビジョン記録を 1 トランザクションで原子化 |
| 6 | リビジョン履歴 | `knowledge_revision`(type,id,rev で追記) |
| 7 | 利用集計・並び | `knowledge_usage` の LATERAL 集計、`verified_at`/`failed`/usage 順のフィード |

Cloud Storage は純粋なオブジェクトストアであり、上のうち **1 以外は
そのままでは提供できない**(クエリエンジンも二次索引も全文/ベクトル
検索もクロスオブジェクトのトランザクションも無い)。したがって問いは
「これらを別の場所でどう賄うか」に帰着する。

## 3. 置換設計: GCS を正本、インメモリを索引

鍵は ochakai が **設計として小さい** ことだ。人間とエージェントが
キュレーションする密度重視のベースで、全体が git 管理可能な OKF バンドル
(Markdown + YAML)に round-trip する。現実的な規模は数百〜数千エントリ、
各数 KB — 合計で単桁 MB。**つまりベース全体がメモリに載る。**

そこで次の構成を採る。

- **正本は GCS**: 1 エントリ = 1 オブジェクト(本番では OKF ドキュメント、
  すなわちバケットがそのまま browsable な OKF バンドルになる)。添付は
  既に GCS(design doc 0013)。
- **索引はインメモリ**: 起動時に全オブジェクトを List + Get して
  `map[id]Knowledge` を構築(コールドスタートの一度きりの O(N) GET)。
- **検索・関係クエリ・並びはすべて Go のインプロセス走査**: PostgreSQL が
  索引で捌いていたものを、メモリ上の線形走査に置き換える。規模が小さい
  ので実用上問題ない。
- **書き込みは GCS のオブジェクト世代(generation)を条件付き**に行う:
  これが 5 の楽観ロックと原子性を代替する中心機構(§4)。

### 3.1 楽観ロックはむしろ強くなる

現状の If-Match は `updated_at` の一致を **アプリが手で** 突き合わせる
(design doc 0025 §11)。GCS は `x-goog-if-generation-match` を **ストレージ
自身が** 強制する。generation 0 = 「存在しなければ作成」(create-only)、
正の値 = 「まさにこの版なら上書き」。手作りの楽観ロックが、ストレージの
ネイティブなプリコンディションに置き換わる — 純粋な改善である。

### 3.2 リンク索引は本文から導出、ゆえに自己修復

ochakai はリンクをフィールドとして保存せず **本文から導出** する
(design doc 0024、`domain.LinksFromBody`)。この 1 点が GCS 単体設計の
成否を分ける。索引の起動時再構築でリンクを常に本文から引き直せば、
**インメモリのリンクグラフは常に「本文が言っていること」と一致する。**
永続化された索引が本文とズレる、という不整合が原理的に発生しない。

## 4. 検証: 実コードで載っているリスクを潰す

本調査では、上記設計の **load-bearing な機構だけ** を実装した検証用
パッケージ [`internal/objstore`](../../internal/objstore) を用意し、
テストで各リスクを実証した(本番の `internal/store` を置き換えるもの
ではない — §6 の限界のとおり全面移行はより大きな作業)。

`ObjectStore` は GCS の世代セマンティクスだけを写した最小インターフェース
で、テストは `internal/blob` と同じくフェイク実装(世代付き、
プリコンディション強制、Put 失敗注入)に対して走るため、実バケットも
Postgres も要らない。

| テスト | 実証した主張 |
|---|---|
| `TestRebuildFromObjects` | GCS のバイトだけから全エントリを復元(コールドスタート走査が DB 接続を代替) |
| `TestCreateIsCreateOnly` | generation-0 プリコンディションが重複 id を弾く |
| `TestOptimisticConcurrency` | 古い `updated_at` を持つ更新が、間に入った書き込みに敗れ `ErrConflict`(design doc 0025 §11 のロックを世代で強制) |
| `TestSearchRanking` / `TestSearchFilter` | trigram + 部分一致フロア + verified ブーストの順位付け、rejected の既定除外と明示照会 |
| `TestBacklinks` | JSONB 包含のバックリンクをインメモリのリンクグラフで再現 |
| `TestModelDefiningMetric` | `attrs.spec.metrics[]` の JSONB パス包含をインメモリ走査で再現 |
| `TestMoveRewritesReferences` | 1 トランザクションのリネーム+参照書き換えを、複数オブジェクト書き込みで実現 |
| **`TestMoveCrashSelfHeals`** | **リネーム後・参照書き換え途中でクラッシュしても、再オープンで「ぶら下がるが可視」な辺が残るだけで索引破壊は起きず、修復再実行で治る**(§3.2 の帰結) |
| `TestUsageAggregation` | 利用集計が join 無しでインメモリに畳める |

`go test ./internal/objstore/` は 10 件すべて成功する。とくに
`TestMoveCrashSelfHeals` が、**トランザクションの不在を「本文からの
リンク導出」で埋められる** ことを具体的に示している点が、この設計の
肝である。

## 5. コスト・運用のトレードオフ

| 項目 | 現状(Cloud SQL) | GCS 単体 |
|---|---|---|
| 月額支配項 | Cloud SQL ~$9–10 | ストレージ + リクエストでセント単位 |
| 書き込み待ち時間 | ~ミリ秒(DB) | ~数十ミリ秒(GCS 往復) |
| コールドスタート | 温かい接続 | 全件走査(数 MB で秒オーダ) |
| 耐久性 | 単一 Cloud SQL インスタンス | GCS(より高い) |
| 楽観ロック | アプリが手作り | ストレージがネイティブ強制(改善) |

コールドスタート走査を避けたければ `min-instances=1` にするが、常時
起動のインスタンスは Cloud SQL より高くつきうる。低トラフィックなら
ゼロスケール + まれなコールドスタート走査の方が安い。**このテンションが
GCS 単体化の実際の設計判断点** である。

## 6. 無理が出るところ(正直な限界)

「出てきたら知らせてほしい」への回答。次の 5 点が、Cloud SQL が
**タダで** くれていたものであり、GCS 単体では自前で背負うことになる。

1. **水平スケールを失う(最大の譲歩)。** 索引と書き込みが 1 プロセスの
   メモリ・mutex に依存するため、**書き込みインスタンスは 1 つに固定
   (`max-instances=1`)** しなければならない。複数インスタンスが各自の
   インメモリ索引に書くと、片方の書き込みがもう片方に見えず、`Move` の
   複数オブジェクト書き換えがインスタンス間で競合して壊れる。
   Postgres は「全インスタンス共通の一貫ビュー + トランザクション」を
   提供する、まさにこの役割を担っていた。小チームのキュレーション用途
   なら許容できるが、高並行が要件になった瞬間に破綻する。GCS ベースの
   ロック/リースで多インスタンス化する道はあるが、脆く、本プロジェクトの
   簡潔さの理念に反する。

2. **検索が索引付きから全走査になる。** クエリごとに O(N)。数千件までは
   快適、数万件超で劣化する。ochakai の密度重視の規模なら問題にならないが、
   「無制限に育てる」設計ではなくなる。

3. **クロスオブジェクトの ACID が無い。** `Move` と多書き込み操作は
   「インプロセス直列化 + 起動時のリンク再導出による自己修復」に頼る
   (§4 で検証済み)。破壊は起きないが、クラッシュ後に「ぶら下がる辺」を
   掃除する起動時リペアの実装が要る。Postgres ではトランザクションで
   タダだった。

4. **コールドスタート走査とゼロスケールのテンション。** §5 のとおり。
   ゼロスケール(安い)を採るとスケールアウト時に多インスタンスが立ち
   うるため、結局 `max-instances=1` が必須になり、リクエストは直列化
   される。用途に合うかは運用判断。

5. **意味検索(pgvector)の置換。** 有効化時、埋め込みのブルートフォース
   コサインをメモリ上で回す。数千ベクトルなら実用的だが、`pgvector` の
   索引が持っていた性質(大規模での近似最近傍)は失う。ochakai の規模では
   許容。

## 7. 移行計画(採用する場合)

段階的に、Postgres 経路を壊さずに進められる。

1. **Store インターフェースの抽出。** 現状 `service.Service` は具象
   `*store.Store` に依存している。まず `internal/store` の公開メソッド群を
   インターフェース化する(純粋なリファクタ、挙動不変)。
2. **`internal/objstore` を本番品質に拡張。** 本検証パッケージを土台に、
   リビジョン履歴、添付メタデータ、`ListByUsage`/`ListByFailed`、Browse
   (GCS の delimiter 一覧がフォルダツリーに直結)、OKF ドキュメント
   としての永続化、起動時リペアを実装。
3. **設定と配線。** `OCHAKAI_DATABASE_URL` を任意化し、`OCHAKAI_GCS_BUCKET`
   のみで起動する経路を追加(現状は添付専用)。`max-instances=1` を
   デプロイ手順の前提として明文化。
4. **利用カウンタの永続化。** インメモリ集計を単一 GCS オブジェクトへ
   定期スナップショット(design doc 0025 §10 の best-effort 性質に合致)。
5. **手順書とコスト表の更新。** Cloud SQL 行を削除、`max-instances=1` の
   制約とコールドスタートの注記を追加。

## 8. 推奨

- **やる価値はある。** コスト(支配項の消滅)と、OKF 正本思想・「小ささ」
  との相性の良さは大きい。バケットがそのまま OKF バンドルになる点は、
  design doc 0003 §3 の「データの可搬性で約束を守る」を一段強くする。
- **ただし §6-1 の単一インスタンス制約を、機能ではなく前提として受け入れ
  られる場合に限る。** これは「小チームの living knowledge base」という
  ochakai の対象読者(README)には合致するが、汎用の高並行サービスを
  志向するなら Postgres を残すべきだ。
- 実装は §7 の段階移行を推奨。本検証パッケージ `internal/objstore` は
  その第 2 段の出発点として、載っている全リスクが潰せることを既に
  実証している。
