# ochakai 設計ドキュメント 0008: 画像添付

Status: Accepted(2026-07-17 の議論で決定)
Date: 2026-07-17

## 1. 目的

ナレッジエントリに画像を持たせる。「正常時のダッシュボードはこう見える」
のスクリーンショット(insight)、ER 図(table)、定義元のスライド(metric)——
「メトリクスの読み方」という ochakai の核となる知識は本来ビジュアルと相性が
良い領域であり、テキストだけでは伝わらない知識の置き場が必要だった。

## 2. 世界観: 画像は単独では意味を持たない

**ほとんどの画像は「画像 + 解説文」のセットで初めて知識になる。** この観察が
設計全体を貫く:

- **検索・取得の単位はエントリ(md)であり、画像は単独の検索結果にならない。**
  `get_knowledge` は body(解説文)+ 添付メタデータを常に一緒に返すので、
  エージェントが画像だけを受け取って解説文を取り逃がす経路が構造的に存在
  しない。画像の alt テキストとキャプションは body に書かれ、既存の trigram
  検索がそのまま効く。
- **metadata-first, fetch-on-demand。** 画像 1 枚は数万トークン相当になり得る。
  search / get / get_context はバイト列を返さずメタデータ(name, media_type,
  size, sha256)だけを返し、バイト列は明示的な取得(`get_attachment` /
  `GET /api/v1/attachments/...` / `ochakai get --download`)で初めて渡る。
  設計ドキュメント 0004 のコンテキスト経済学の画像版である。
- **No LLM inside は画像でも変わらない。** OCR・キャプション生成・画像の意味
  検索はしない。画像を解釈し、読み取った内容を `update_knowledge` で body に
  書き戻すのはクライアントエージェントの仕事——それが正しい write-back
  ループである。

## 3. 決定

### 3.1 ストレージ: PostgreSQL、コンテンツアドレス方式

バイト列は SHA-256 でアドレスされる不変の `blob` 行に住み、`attachment` 行が
エントリ + ファイル名 → blob を対応づける。

- **Postgres 一本を維持する。** GCS を足せば「Docker だけでローカル起動」
  「バックアップは Cloud SQL 一箇所」「secret-zero に新しい設定を足さない」の
  すべてが壊れる。想定サイズ(スクリーンショット × 数百エントリ)なら
  問題なく、限界が来たら blob テーブルだけを外部ストレージへ逃がす移行路が
  sha256 キーのおかげで閉じている。
- 同一画像の重複は自動排除され、リビジョンはハッシュで歴史上の内容を指せる。
  blob は削除しない(リビジョン同様、歴史は保持する)。
- 添付・削除はエントリへの変更として扱う: `updated_at` が進み、変更後の
  添付リストをスナップショットに含むリビジョン(change: `attach` / `detach`)
  が残る。

### 3.2 制約

| 項目 | 値 | 理由 |
|---|---|---|
| media type | png / jpeg / gif / webp(バイト列からスニッフィング、申告は信用しない) | SVG はスクリプトを持てるため除外——API から配信すれば全ナレッジ著者に Web UI への XSS 経路を渡すことになる |
| サイズ | 5 MiB / 枚 | スクリーンショット・図の想定。原本置き場ではない |
| 個数 | 20 / エントリ | 暴走ガード |
| 名前 | 1 パスセグメント、`*.md` 禁止 | バンドルパス・URL にそのまま埋まる。`.md` はコンセプト文書に化けられるため禁止(index.md / log.md 予約も同じ規則から導かれる) |

### 3.3 API 三面

- **REST**: `PUT / GET / DELETE /api/v1/attachments/{type}/{id...}/{name}`。
  添付名は 1 セグメントなので「最後のセグメントが常にファイル名」という分割に
  曖昧さがない。`/usage` と同じ理由で `/knowledge/` の外に置く(階層 ID の
  `{id...}` の後にサフィックスは置けない)。GET の ETag は sha256。
- **MCP**: `get_attachment` が image content block(base64)+ メタデータを返す。
  `get_knowledge` の説明文が添付メタデータの存在と「見てから書き戻す」習慣を
  エージェントに教える。
- **CLI**: `ochakai attach <type>/<id> <file...>` / `detach` /
  `get --download <dir>`。Claude Code はダウンロードされたファイルを Read する
  だけで画像を認識できるため、シェルを持つエージェントにはこちらが本命。

### 3.4 OKF バンドルとの往復

OKF 仕様は非 markdown ファイルについて完全に沈黙している(禁止も規定もなく、
予約はファイル名 `index.md` / `log.md` のみ。コンシューマは未知のコンテンツに
寛容であることを要求され、リンク切れも許容しなければならない)。この世界で
一貫性を保つ規則は 3 つ:

1. **帰属は参照駆動、パス非依存。** インポート時、画像がどのエントリの添付かは
   concept 文書の body 中の markdown リンク(`![...](path)` / `[...](path)`)を
   解決して決める。相対リンクは文書のディレクトリ基準、`/` 始まりはバンドル
   ルート基準(いずれも SPEC §5 の形式)。どこに置かれていても構わない——
   他プロデューサのレイアウトは予測不能であり、実際 OKF 公式の GA4 バンドルは
   `viz.html` をルート直下に置く。参照されていない非 md ファイルは従来どおり
   スキップ + レポート。
2. **エクスポートの正準レイアウトはエントリ同名ディレクトリ。**
   `<type>/<id>/<name>` が `<type>/<id>.md` の隣に並ぶ。OKF の階層 ID 思想と
   同型(「このエントリに関するすべては、このエントリ名の名前空間に住む」)で、
   `_assets` のような語彙の発明を避ける。body からの正準リンクは
   `![alt](<id 末尾セグメント>/<name>)` で、GitHub がそのままレンダリングする。
   階層 ID の子エントリと同じディレクトリに同居しても無害(concept パーサは
   非 md を見ない、index.md 生成はエントリツリーだけを見る)。
3. **外部由来のファイルは元の場所に帰る。** インポート時にバンドル内パスを
   `okf_path` として保存し、エクスポートは正準レイアウトより優先してそこへ
   書き戻す。元の body リンクが切れず、バイト同一往復に近づく——`okf_type` と
   同じ「元の表記を保存して往復」パターンの画像版。

リンク形式のトレードオフ: 仕様の推奨はルートアブソリュートだが、GitHub の
レンダリングでは `/` 始まりが壊れる。「plain markdown が git で幸せに暮らす」
原則を優先し、正準リンクは相対で出し、インポートは両形式を解決する。

### 3.5 リネームとの関係

エントリのリネーム(export → 編集 → import)ではディレクトリ名が付いて回るため
body 内リンクの書き換えが要るが、リネームがもともと手作業フローである以上
新しい負担ではない(設計ドキュメント 0005 §4)。

## 4. やらないこと

- **画像の解釈・OCR・キャプション生成**(§2 のとおり、クライアントの仕事)。
- **画像そのものの意味検索。** alt / キャプション経由のテキスト検索で始める。
  将来 Vertex のマルチモーダル埋め込み(既存のテキスト埋め込みと同じく
  オプトイン・ADC 認証)を足す拡張点は残るが、その場合もヒットは親エントリに
  解決する——画像を単独の検索結果にしない原則は変わらない。
- **SVG。** スクリプトを持てる形式は許可リストに入れない。
- **GCS 等の外部ストレージ**(§3.1 のとおり、移行路だけ残して今はやらない)。
- **create/update ペイロード経由の添付。** 添付は専用エンドポイントの仕事で、
  エントリの JSON に base64 を混ぜない(検索インデックスとリビジョンが
  肥大化する)。
