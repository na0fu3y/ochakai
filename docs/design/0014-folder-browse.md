# ochakai 設計ドキュメント 0014: フォルダブラウズ

Status: Accepted(2026-07-18 の議論で決定)
Date: 2026-07-18

## 1. 目的

エントリが増えると、検索だけでは「何があるか」が見えなくなる。ID は
設計ドキュメント 0005 以来スラッシュ区切りの階層スラグであり、OKF
エクスポートはそれをそのままディレクトリツリーとして書き出す — つまり
階層はすでにデータにある。無かったのは人間がそれを歩く画面と、その
画面を一階層ずつ支える API だけである。本ドキュメントで両方を足す。

## 2. 決定

### 2.1 API: 一階層ずつ返す `GET /api/v1/browse`

- `type` なし: 生きているエントリを持つ型の一覧と件数(ツリーの根)。
- `type` あり(+ 任意の `prefix`): その位置の**直下**にあるサブ
  ディレクトリ(配下件数つき)と、その階層のエントリ(title / status /
  updated_at の軽量射影 — body は返さない)。
- rejected と soft-delete 済みは出さない(検索のデフォルトと同じ)。
- 一階層のエントリは 1000 件で打ち切り、`truncated` で申告する。
  ページングはしない — そこまで平たいディレクトリは分割すべきモデリング
  の問題であって、ページャで歩く問題ではない。
- prefix 照合は LIKE ではなく文字列比較(`left(id, …) = prefix`)。
  ID には `_` が使えるので、LIKE ではワイルドカードに化ける。
- ブラウズは検索ではない: usage は記録しない。木を歩いて何があるかを
  見ることは、検索ヒットが測る需要シグナルではない。

### 2.2 UI: Browse タブ

Web UI に Explore と並ぶ Browse タブを足す。根に型、下に ID セグメント
のディレクトリ、葉にエントリ(タイトルとステータスバッジ、クリックで
詳細へ)。ノードは開いたときに一階層だけ読み込み、開閉状態はセッション
内のナビゲーションを跨いで保たれる。単一ファイル・ビルドなしの方針
(設計ドキュメント 0006)は変えない。

### 2.3 REST のみ。MCP ツールにはしない

ブラウズは「人間が眺めて把握する」ための面であり、エージェントには
search / get_context という目的特化の入口がすでにある(README の
「Web UI is the human half of the loop」)。REST が MCP の上位集合で
あるという既存の整理(設計ドキュメント 0007)にもそのまま収まる。

## 3. やらないこと

- **ディレクトリの実体化。** ディレクトリは ID の接頭辞に過ぎず、
  行を持たない。空ディレクトリは存在せず、リネームはエントリ ID の
  変更である。
- **ツリーの一括取得 API。** 深い木を一度に返すと応答サイズが読めない。
  一階層ずつで、UI の体感には十分。
- **並べ替え・フィルタ。** ブラウズは名前順の安定表示に徹する。
  絞り込みたければ Explore(検索)がある。
