# ochakai 設計ドキュメント 0016: 配置とタイプの分離 — パスが住所、タイプは属性

Status: Accepted(2026-07-19 承認。破壊的変更は許容、互換シムは置かない方針)
Date: 2026-07-19

## 1. 目的

現状のナレッジの住所は `<type>/<id>` の 2 部構成で、バンドルパスの
第 1 セグメントは必ずタイプである(設計ドキュメント 0005 §3.1)。
この「最上位はタイプで割る」という固定が、次の 3 つの問題を生んでいる。

1. **ドメインの散逸。** 「sales」に関する metric・query・table は
   `metric/` `query/` `table/` に散らばり、LLM や人間が「近い
   ドキュメント」として一括で辿れない。ID 階層をタイプ配下で揃える
   ことはできるが、横断は index 上では見えない。
2. **バンドル境界の消失。** 外来バンドルを import すると、その第 1
   セグメント(`tables/` 等)がタイプに合流する。ga4 と stackoverflow を
   両方取り込めば互いの `tables/` が混ざり、どのバンドル由来かは
   パスから消える。単一ディレクトリ包みの自動アンラップ(0005 §3.3)は
   バンドル名=包みディレクトリ名を**意図的に捨てる**規則であり、
   失われている次元がそこに最も明確に現れている。
3. **配置と振る舞いの二者択一。** タイプは自由スラグなので
   `ga4/tables/orders.md`(type=`ga4`)とドメイン先頭に置くことは今日でも
   できるが、そうすると type=`metric` に結びつく compile_sql、`query` の
   ゴールデンクエリカナリア、`table` の resource エクスポートが付かない。
   「ドメインでまとめる」か「タイプの振る舞いを得る」かを選ばされる。

本ドキュメントは、この固定を外し、OKF の「concept ID = パス、レイアウトは
規定しない」という自由な構造をそのまま ochakai の構造にする。破壊的変更は
許容し、互換レイヤーは置かない(レガシーの温床になるため)。

## 2. 背景: knowledge-catalog の実物の観察

OKF の本家 [knowledge-catalog](https://github.com/GoogleCloudPlatform/knowledge-catalog/tree/main/okf/bundles)
の `okf/bundles/` 直下は `crypto_bitcoin` / `ga4` / `stackoverflow` —
**トップレベルはデータソース(ドメイン)単位**である。一方 `ga4` バンドルの
内部は `datasets/` `references/` `tables/` と、タイプ風のディレクトリで
切られている。つまりタイプ分け自体が悪いのではない。あちらでは

- **バンドル = LLM が一括で読む単位**がドメインで切られ、
- タイプ風のディレクトリは**各バンドル内のローカルな規約**にすぎない
  (SPEC はタクソノミーもレイアウトも規定しない。type は frontmatter の
  必須キー、concept ID = パス、それだけ)。

ochakai はナレッジベース全体が単一バンドルであり、その最上位をタイプで
固定している。欠けているのは「タイプより上の階層」であって、タイプ
ディレクトリという慣習ではない。

## 3. 世界観: 場所は利用者のもの

設計ドキュメント 0005 §2 の「推奨はするが、強制しない」はタイプの
**語彙**に適用された(5 タイプは推奨、自由タイプは一級市民)。本ドキュメント
は同じ原則を**配置**に適用する。

- タイプは「それが**何であるか**」であり、「それが**どこにあるか**」ではない。
  何であるかは frontmatter が言い、どこにあるかはパスが言う。両者は直交する。
- どこに置くかは知識のモデリングであり、利用者のものである。タイプ先頭
  (`metric/sales/...`)もドメイン先頭(`sales/orders.md` に `type: metric`)も
  一級であり、サーバーはどちらも強制しない。
- 「Your knowledge is yours」の帰結: 外来バンドルは元の形のまま、
  自分の選んだ場所に入り、その場所のまま出て行ける。

## 4. 決定

### 4.1 ID はバンドル・ルートからのフルパス(唯一の住所)

concept ID = ファイルパスから `.md` を除いたもの、という OKF の規則を
そのまま ochakai の主キーにする。

- ID はスラッシュ区切りのスラグ列(セグメント規則は 0005 §3.1 のまま)。
  第 1 セグメントに何の意味も与えない。ルート直下の 1 セグメント ID も
  正規のエントリである。
- DB の主キーは (type, id) 複合から **id 単独**になる。同じパスに 2 つの
  エントリは存在できない(ファイルシステムと同じ)。type は列として残る
  (NOT NULL・1 セグメントスラグ・自由)。検索フィルタ・タイプ結合の
  振る舞い・表示は従来どおり type 列で動く。
- 予約する末尾セグメントに、`index` に加えて **`log` を追加**する。
  現状、末尾 `log` の ID は log.md として export され import で黙って
  スキップされる(既知の往復欠陥)。本改訂で構文的に塞ぐ。

### 4.2 タイプは frontmatter の属性。推測はしない

- **エクスポート**: パスは `<id>.md`。frontmatter の `type` は従来どおり
  表示名写像(`Metric` / `Golden Query` …、自由タイプは okf_type の綴り
  保存)。**`id` は frontmatter に出力しない** — パスが正であり、現状は
  外来バンドルに `id:` 行を書き足して byte 同一の往復を壊している。
- **インポート**: frontmatter の `type` が唯一の情報源(typeFromOKF の
  写像そのまま)。type は OKF SPEC でも必須キーであり、無いファイルは
  concept ではない — **推測せず、スキップしてレポートする**(非 Markdown
  ファイルと同じ扱い)。パスからタイプを推測する規則は全廃する。
- 帰結: ga4 バンドルの `tables/orders_.md`(`type: Table`)は位置
  `tables/orders_` のまま **type=`table` として取り込まれ、resource
  エクスポート等の振る舞いが付く**。§1-3 の二者択一が消える。

### 4.3 アンラップ廃止: tar に入れた形がそのまま構造

アーカイブの自動アンラップ(StripWrapper)と `--keep-root` フラグを
**削除**する。あの機構は「第 1 セグメント = タイプ」の世界で包み
ディレクトリ名が全エントリのタイプに化けるのを防ぐ防御であり、パスが
タイプを主張しなくなった時点で防御対象が消滅する。

- `tar czf ga4.tgz ga4/` は `ga4/` 配下に取り込まれる — **バンドル =
  名前空間ディレクトリ、が既定の(そして唯一の)動作**になる。§1-2 の
  バンドル境界の消失は、フラグではなく既定動作で解決する。
- `ochakai export` の tarball はルート直で、往復は無変更。
- 見たままが構造(WYSIWYG)。通知も抑止フラグも要らない。

### 4.4 移行: 新 id = `<type>/<id>`

既存エントリの新 ID は旧 type と旧 id をスラッシュで結合したものとする。
type にスラッシュは入れないので単射であり、衝突しない。これは互換の
ためではなくデータ保全である — が、この綴りを選ぶと、直列化された
すべての表現(エクスポートパス `<type>/<id>.md`、URI
`ochakai://<type>/<id>`、リンクの target `metric/revenue`)が既に
この形なので、**既存データの参照文字列は書き換え不要で新世界の id として
読める**。type と id が別フィールドである場所は DB 列・MCP ツール引数・
frontmatter の 3 箇所しかない。

マイグレーション 0010 で knowledge / revisions / usage / attachments の
id を書き換え、主キーから type を外す。エントリの再配置はしない —
既存データは従来のタイプ先頭レイアウトにそのまま残り、動かすのは
利用者である。

### 4.5 各サーフェスの変更(0015 の役割ごと、シムなし)

- **REST**: ルートは `{id...}` 一本(`/api/v1/knowledge/{id...}`、
  `/usage/{id...}`、`/revisions/{id...}`、`/backlinks/{id...}`)。
  0005 §3.5 が残した `/knowledge/{type}/{id}/usage` の**レガシー
  エイリアスは削除**する。書き込みボディの `type` は create / update とも
  必須(格納値からの補完はしない)。
- **MCP**: 住所を取るツール(get_knowledge / get_attachment /
  get_knowledge_usage / delete_knowledge)は引数を **id 一本**にする
  (ツールスキーマはコンテキスト予算 — 0015 §2)。create / update は
  メタデータとして type を残す。report_outcome の target は id。
  リソーステンプレートは `ochakai://{+id}`。
- **CLI**: ref は id そのもの。browse の引数は prefix 一本(下記)。
  import から `--keep-root` が消える(§4.3)。
- **Web UI**: Browse の根がタイプ一覧からトップレベルセグメント一覧になる。
  タイプはエントリのバッジとして表示する。
- **URI**: `ochakai://<id>`。

### 4.6 インポート規則の改訂(0005 §3.3 との差分)

| 事象 | 0005 | 本ドキュメント |
|---|---|---|
| パスと frontmatter の type 不一致 | パスが勝つ | 不一致という概念が消える(パスは type を主張しない)。okf_type は表示綴りの保存として存続 |
| ルート直下の concept | 再エクスポートで type ディレクトリ配下に移動 | その場に留まる(往復が安定する) |
| type 無しの frontmatter | 第 1 セグメントが type | スキップしてレポート(SPEC 違反。推測しない) |
| 単一ディレクトリに包まれたアーカイブ | 自動アンラップ + 通知、`--keep-root` で抑止 | 機構ごと削除。包みディレクトリはそのまま名前空間になる |

その他の規則(index.md / log.md のスキップ、隠しファイル、非 Markdown、
改行コード、サーバー所有キー)は変更しない。

### 4.7 ブラウズの改訂(0014 との差分)

`GET /api/v1/browse` のパラメタを `type` + `prefix` から **`prefix` 一本**に
する。prefix なしは木の根 = トップレベルセグメントの一覧(件数つき)。
エントリ射影に type を一列足す(UI バッジ・CLI 表示用)。それ以外の決定
(一階層ずつ、1000 件打ち切り、usage を記録しない、ディレクトリは
実体化しない)は変更しない。

### 4.8 推奨は語彙まで。配置には原則だけ

5 タイプの推奨(語彙と振る舞い)は変えない。配置についてサーバーも
ドキュメントも既定を押し付けず、原則を 1 行だけ示す:
**「LLM に一緒に読ませたいものを同じディレクトリに置く」**。
ドメイン先頭・タイプ先頭・外来バンドルの原型維持、いずれもこの原則の
正当な適用である。エクスポートの index 生成もこれに合わせ、ルートで
組み込みタイプを先頭に並べる特例(タイプ先頭レイアウトの名残)を
やめて**全階層アルファベット順**にする。

## 5. やらないこと

- **バンドルの一級エンティティ化。** 名前空間テーブル、バンドル単位の
  ACL・一括操作は作らない。境界はディレクトリ(= id の接頭辞)で表現し、
  行を持たない(0014 §3「ディレクトリの実体化」と同じ理由)。
- **リネーム / ムーブ API。** 従来どおり export → 編集 → import で行う。
  本改訂で再配置の需要は上がるが、provenance の付け替え問題
  (issue #42)とセットで別ドキュメントの仕事とする。
- **既存データの自動再配置。** マイグレーションは綴りを保存するだけ。
  ドメイン先頭へ組み替えるかは利用者の判断。
- **タイプの階層化・複数化。** type は 1 セグメントスラグ 1 個のまま。
  分類を増やしたければ tags と場所がある。
- **手書き index.md の保存。** 外来バンドルのキュレーションされた
  index.md は取り込まず、エクスポートで再生成する。index は導出物と
  いう整理(0005 §3.3)を維持する。エントリの title / description が
  index に反映されるので、キュレーションは各エントリ側で行える。
- **移行用の互換シム。** 旧 2 引数 MCP スキーマの受理、REST レガシー
  エイリアス、PUT の type 補完、いずれも作らない。

## 6. 検討した代替案

- **案 B: 現状維持 + ID 規約**(タイプ配下でドメイン階層を揃えることを
  規範化)。ドメイン横断は index からは見えないままで、バンドル境界の
  消失(§1-2)も解決しない。強制されない規約は drift する。却下。
- **案 C: バンドル / 名前空間の一級化**(バンドルをテーブルに持ち、
  import 単位で管理)。DB・API・認可に新しい次元が入る。OKF はバンドル間
  参照を規定せず、カタログ本家もディレクトリ以上の構造を持っていない。
  過剰。却下。
- **案 D: パスのセグメントから type を推測**(`tables/` を見つけたら
  type=table、親ディレクトリ名をフォールバックにする等)。当初案には
  「frontmatter に type が無ければ直近の親ディレクトリ名」という
  フォールバックがあったが、type は SPEC 上必須であり、推測はマジックで
  ある。frontmatter に書けば済む。全面的に却下し、無ければスキップ。

## 7. 壊れるもの(許容済みの棚卸し)

破壊的変更を許容する方針を受け、シムは置かない。壊れるものを列挙する。

- **MCP の住所系ツールのスキーマ**: type + id の 2 引数 → id 1 引数。
  エージェント側の設定・プロンプト(CLAUDE.md 等)は更新が必要。
- **REST のレガシー usage エイリアス**(0005 §3.5): 削除。
- **PUT / create の type 必須化**: ボディに type の無い書き込みはエラー。
- **アンラップの廃止**: 包み付き tar の import 結果が変わる(包みが
  名前空間として残る)。
- **type 推測の廃止**: frontmatter に type の無いファイルは import されず
  レポートに載る(従来は第 1 セグメントで受けていた)。
- **index の並び順**: ルートの組み込みタイプ優先が消え、全階層
  アルファベット順になる(旧 export と diff が出る。`id:` 行の削除も同様、
  1 回きり)。

なお移行(§4.4)の綴り選択により、エージェントや外部システムが保持する
**参照文字列**(URI・リンク target・エクスポートパス)は書き換えなしで
新世界でも同じエントリを指す。壊れるのはスキーマであって、参照ではない。

リリースは 0.10.0 に一括で載せる(pre-0.9.0 の retract 済み方針により
0.9.x 系の利用者のみを想定すればよい)。
