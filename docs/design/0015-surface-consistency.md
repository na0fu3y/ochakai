# ochakai 設計ドキュメント 0015: サーフェス一貫性の方針

Status: Accepted(2026-07-18 の議論で決定)
Date: 2026-07-18

## 1. 目的

ochakai には 4 つのサーフェスがある: REST(/api/v1)、MCP(/mcp)、
CLI、Web UI。機能を足すたびに「どのサーフェスに載せるか」を都度判断
していると、載せ忘れ(PR #71 で browse が CLI に入らなかった)や
文言の乖離(設計ドキュメント 0013 の添付一般化がヘルプに反映されな
かった)が繰り返される。本ドキュメントは各サーフェスの役割と、
**意図して実装しないもの**を明文化し、次の機能追加が参照する場所を
作る。

## 2. 各サーフェスの役割

- **REST は唯一の契約。** すべての機能はまず /api/v1 に載る(または
  どこにも載らない)。他の 3 サーフェスは全部そのクライアントである
  (設計ドキュメント 0007)。上限・enum・デフォルトなどの契約は
  サーバー側に一元化する。
- **MCP はエージェントの目的特化入口。** ツールスキーマはエージェント
  のコンテキストを消費するから、ツール数は予算である。REST の 1:1
  ミラーは目指さない。「エージェントのループが 1 コールで必要とする
  もの」だけを載せる(get_context がその極致)。
- **CLI は完全性のサーフェス。** /api/v1 の純クライアントとして REST
  の全操作をカバーする。シェルを持つエージェントと人間の両方が使う。
  バルク操作(OKF import)は CLI が既存 API を合成して実現する。
- **Web UI はループの人間側。** キュレーション面であって BI ツール
  ではない。検索・ブラウズ・レビュー(verify/reject)・履歴の閲覧
  まで。ページは認証を知らない(設計ドキュメント 0006)。

## 3. 実装しないもの(サーフェス別)

### 3.1 MCP に載せないもの

- **browse。** 設計ドキュメント 0014 §2.3 のとおり。木を歩くのは
  多ラウンドトリップ型の探索で、get_context の「1 コールで全部」と
  正反対。
- **revisions。** エージェントが必要とする信頼シグナル(created_by /
  verified_by / updated_at)はエントリ本体に載っている。全スナップ
  ショット付き履歴はトークン的に重く、監査は人間の仕事。
- **backlinks。** get_context がリンクを逆向きにも展開して同梱済み。
  単独ツールは機能重複になる。
- **添付の書き込み(attach / detach)。** base64 をツール引数に載せる
  のはトークンの浪費であり、思想にも反する: エージェントの書き戻しは
  検索可能なテキストであるべきで、「添付から学んだら本文に書け」と
  get_attachment の description が既に誘導している。
- **バルク(export / import / import-ossie)。** 応答サイズが読めず、
  tar.gz はツール結果に載らない。管理タスクは CLI / CI の領分。

### 3.2 REST に載せないもの

- **SQL 実行。** ochakai はウェアハウス資格情報を持たない。コンパイル
  まで、実行はクライアント。
- **LLM を使う機能(要約・チャット・自動検証)。** LLM を使わない
  ことが決定的コンパイルと人間検証への信頼の根拠。
- **認証・ユーザー管理。** 認証は Cloud Run IAM の仕事(設計
  ドキュメント 0002)。ユーザー DB を持った瞬間 secrets ゼロ運用が
  壊れる。
- **Web UI の配信。** ページを持たないサーバー(設計ドキュメント
  0006)。認証境界をプロキシに置く前提。
- **バルク OKF import(POST /import/okf)。** import は CLI が
  create / update / attach を 1 件ずつ叩いて合成する。件別の
  created / updated / unchanged レポート・dry-run・部分失敗の位置
  特定・actor 記録がすべて既存 API の意味論に乗り、サーバー側バルク
  という「同じ成果物への 2 経路目」(0007 §2.1 の drift の温床)を
  作らずに済む。見直す条件は、CLI を持たないクライアントが import を
  実際に必要としたとき、だけ。

### 3.3 CLI に載せないもの

- **DB 直結コマンド。** `serve` のみ(設計ドキュメント 0007 で決着)。
- **レビューキューのヒューリスティック再実装。** Web UI の stale 判定
  (14 日)や weak-score 閾値は表示層のヒューリスティックであり、
  複製すると二重実装 drift が始まる。CLI は素材の提供
  (`search --sort usage --status draft`)まで。
- **TUI・対話モード。** パイプで合成できる行指向出力が CLI の価値。
  対話が欲しければ `ochakai ui` が立つ。
- **ローカルキャッシュ・オフラインモード。** スナップショットは
  export が担う。キャッシュは鮮度と provenance の問題を持ち込む。

### 3.4 Web UI に載せないもの

- **report_outcome(worked / failed ボタン)。** outcome は「知識に
  基づいて SQL を実行した結果」の証跡であり、SQL を実行しない UI に
  ボタンを置くと行動の裏付けのない worked / failed が混入して再検証
  フラグの信号が汚染される。表示のみが正しい。人間の判断は verify /
  reject という別の語彙で記録される。
- **context ビュー。** エージェント向けのパッキング面。UI に置くと
  「質問を入れる箱」になり、チャット / BI への滑り坂。デバッグは
  CLI で足りる。
- **SQL 実行・チャート・チャット。** README の宣言どおり、エージェント
  に食わせる側に徹する。
- **OKF import アップロード。** バルク REST を作らない判断(§3.2)の
  帰結。複数ファイル・dry-run・件別レポートはブラウザのアップロード
  1 発と相性が悪い。
- **ログインフォーム。** 認証はプロキシの仕事。ページが認証を知らない
  から同じ 1 枚が 2 通りに配布できる(設計ドキュメント 0006)。

## 4. 境界上の判断(現状維持、見直し条件つき)

- **CLI の verify / reject 糖衣コマンド。** status 遷移は update の
  一形態であり、専用動詞は同じ成果物への 2 経路目になるから作らない。
  ターミナルからのレビューが実運用で頻発するなら、**新 API を作らず
  PUT の糖衣として**足すのは許容する。
- **バルク OKF import。** §3.2 のとおり。
- 新しいエンドポイントを /api/v1 に足すときは、本ドキュメントに
  照らして「CLI に載せる(原則 yes)/ MCP に載せる(原則 no、
  エージェントのループが 1 コールで必要とする場合のみ yes)/
  Web UI に載せる(人間のキュレーションに要るなら yes)」を PR で
  明示する。
