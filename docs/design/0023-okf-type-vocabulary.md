# ochakai 設計ドキュメント 0023: 型の語彙を OKF に一本化する

Status: Accepted(2026-07-20)
Date: 2026-07-20

## 1. 目的

同じ 8 つの概念に、ochakai は今 2 つの語彙を持っている。内部スラグ
(`tables`)と OKF 表示名(`BigQuery Table`)である。両者は
`internal/okf` の 4 つの機構 — `okfType` 対応表、`typeAlias`、
`slugifyType`、`AttrOKFType` — で橋渡しされ、**入出力のたびに 2 回変換
される**。この二重語彙は次の 3 つの理由で不要になっている。

1. **元の根拠が 0017 で消えている。** 0016 が複数形スラグを選んだのは
   「knowledge-catalog リファレンスバンドルのディレクトリ配置に合わせる
   ため」だった。しかし 0017 でパスは ID からのみ決まるようになり、
   `internal/okf/bundle.go` は「frontmatter が型の唯一の出所であり、パスは
   もはや型を主張しない」と明示している。エクスポートも `files[k.ID+".md"]`
   で、**型がディレクトリ名を決めることは一切ない**。README の
   「複数形スラグはディレクトリ配置を合わせるため」という説明は陳腐化した
   記述である。
2. **変換が不可逆で、そのための保存ハックが要る。** `slugifyType` は
   `Data Contract` を `data-contract` に潰し、潰した原表記を戻すためだけに
   `okf_type`(`AttrOKFType`)という拡張キーが存在する。往復を成立させる
   ための機構が、往復を壊す機構の後始末をしている。
3. **OKF はタクソノミを登録しない。** 型の値は綴りがそのまま意味である
   自由文字列だと SPEC は定めている。ochakai が別綴りを内部に持って翻訳する
   のは、OKF が意図的に定めなかったタクソノミを ochakai が発明している
   ことに等しい。

## 2. 世界観: 型は OKF の語彙そのもの

0017 は「パスが住所、タイプは属性」とした。本ドキュメントはその属性の
**語彙**についての続きである — **タイプ属性の値は OKF の綴りそのもので
あるべきで、翻訳層を挟むべきではない**。

0016 が「ochakai の推奨型を OKF 表示名にどう見せるか」を決めたのに対し、
本改訂は「見せ方ではなく、それが値そのものである」に進める。0016 の
方向は正しく、最後まで行っていなかっただけである。

`domain.ValidType` の現在のコメントは、型がスラグである理由を
「filters・frontmatter・tool 引数で*喋られる* ochakai の語彙であって、
ファイルパスではない」(0019)と説明している。この理由づけ自体は正しく、
本改訂はそれを維持する — 変えるのは**語彙の中身が ochakai 独自か OKF か**
だけである。型が住所の一部でない以上、スラグである構造的必然はもう無い。

## 3. 決定

### 3.1 内部型 = OKF 表示名(全 8 型)

| 現在の内部スラグ | 新しい内部型 |
|---|---|
| `models` | `Semantic Model` |
| `metrics` | `Metric` |
| `queries` | `Golden Query` |
| `insights` | `Insight` |
| `terms` | `Glossary Term` |
| `datasets` | `BigQuery Dataset` |
| `tables` | `BigQuery Table` |
| `references` | `Reference` |

**全 8 型を同時に移す。** `datasets` / `tables` だけを表示名にすると
残り 6 型の読み替えが残り、語彙が混在して現状より悪くなる。全部か、
ゼロかである。

### 3.2 Type は 1 行の自由文字列になる

`typeRe`(`^[a-zA-Z0-9][a-zA-Z0-9._-]{0,127}$`)を撤廃し、`ValidType` は
次だけを課す:

- トリム後に非空
- 改行・制御文字を含まない
- `/` を含まない(住所と読み違えられないため)
- 128 バイト以内

型は 0022 の意味での**比較される鍵**(検索フィルタの照合対象)なので、
ID・添付名と同じく `domain.Normalize`(NFC)を保存・照合の両方に適用する。
外来バンドルが非 ASCII の型を持ち込みうる以上、0022 の不変条件をここにも
広げる。

### 3.3 照合は大小文字非依存、保存は原表記

インポートは今も `EqualFold` で表示名を受けている。この寛容さを型の
**すべての入口と照合**に広げる:

- 保存は書かれた綴りのまま(`BigQuery Table`)
- 検索フィルタ・CLI `--type`・MCP 引数の照合は大小文字非依存
  (`--type "bigquery table"` が一致する)

これは CLI で空白入りの型を打つ負担への実質的な緩和でもある。

### 3.4 消える機構

- **`okfType` 対応表**(8 エントリ)— 恒等になる
- **`slugifyType`** — 潰す必要がない
- **`AttrOKFType` / `okf_type` 保存**(`internal/okf/okf.go`)— 原表記が
  そのまま型なので、保存すべき「失われた綴り」が存在しない。バンドルが
  文字通り `okf_type` キーを持っていた場合は、ただの拡張キーとして
  attrs に入る(SPEC §4.1 の一般則に戻る)
- **`displayType` / `typeFromOKF`** — 恒等になり、関数ごと消える
- **`typeAlias`**(§3.5)

**インポートとエクスポートは型について恒等になる。** これが本改訂の
主たる見返りであり、消えるのは対応表 1 つではなく 5 つの機構である。

### 3.5 `typeAlias` は撤廃する

v0 系は互換のためのコードを持たない方針である。`typeAlias`(0016 以前の
単数形スラグ、複数形スラグ、汎用 `Table` / `Dataset`)は**丸ごと削除**する。

結果として **`internal/okf` から型の読み替えが完全に消え、インポートも
エクスポートも型について恒等になる**。旧綴りの型を持つ外来ドキュメントは
エラーにならず、その綴りのままの自由型として取り込まれる(`type: tables`
は自由型 `tables` になる)— 0005 の「自由型は第一級」がそのまま働くためで、
ここに特別扱いは要らない。

保存済みデータの綴りは §3.6 のマイグレーションで一度だけ書き換える。
これはコード上の互換層ではなくデータ移行であり、v0 方針と矛盾しない。

### 3.6 マイグレーション 0014

0011 が **型を全テーブルのキーとパスから外している**ため、0010 の
複数形化と違って移行はごく小さい:

- `knowledge.type` の 8 値を UPDATE
- `knowledge_revision.snapshot->>'type'` を UPDATE
- **リンク target・URI・ID の書き換えは不要。** 0017 以降 `URI()` は
  `ochakai://<id>` で型を含まず、link target は ID(バンドルパス)である。
  0010 が必要とした正規表現によるリンク書き換えは、ここでは丸ごと不要。
- `knowledge_revision` / `knowledge_event` / `knowledge_usage` /
  `attachment` / `knowledge_embedding` は 0011 で型列を落としているので
  対象外。
- CHECK 制約は 0003 で撤廃済みなので触るものがない。
- **主キー衝突は起こり得ない。** 0010 は主キーが `(type, id)` だったため
  自由型との衝突でロールバックしうると警告していたが、0011 以降 `id` が
  単独主キーなので、型のリネームは衝突を作らない。
- **`attrs.okf_type` は型に畳み戻してから属性を落とす。** この属性は旧
  slugifier が潰した原表記(`Data Contract` を型 `data-contract` として
  保存)の置き場であり、**エクスポートはこれを type として書いていた**。
  型列に昇格させることは、旧エクスポートの出力を再構成ではなく保存された
  値にすることに等しい。これを先に走らせるので、原表記を持つエントリは
  スラグ改名の対象から自然に外れる — 著者の綴りが優先されるべきなので
  正しい順序である。
- 自由型は(`okf_type` の復元を除き)触らない。
- **再エンベディング不要。** `embeddingText`
  (`internal/service/service.go`)は id・title・description・tags・
  question・body から作られ、**型を含まない**。0020 / 0022 と違って
  既存ベクトルは有効なままである。

### 3.7 表層の更新

型語彙を列挙している箇所を新表記に揃える:

- `cmd/ochakai/completion.go`(zsh / bash / fish の 4 箇所)
- `internal/mcpserver/mcpserver.go`(tool 説明と jsonschema 4 箇所)
- `api/openapi.yaml`(説明と examples)
- `internal/webui/index.html`(`TYPE_ICONS` のキー、新規作成フォームの
  ヒント)
- `README.md` の型表 — あわせて「複数形スラグはディレクトリ配置を
  合わせるため」という **0017 で陳腐化した理由づけを削除する**(§1)

## 4. やらないこと

- **ID・パスの変更。** 住所は 0017 のまま。型は住所の一部ではない。
- **リンク target・URI の変更。** 型を含まないので本改訂と無関係。
- **型を閉じた集合にすること。** 自由型は 0005 のまま第一級であり、
  `ValidType` の緩和はむしろ自由型の表現力を上げる。
- **DB の CHECK 制約の復活。** 検証はアプリ側(0003)のまま。
- **`title` と同じ「省略時フォールバック」の型版。** 型は OKF が required
  と定める唯一のキーであり、0017 の「type 無しはスキップ、推測しない」を
  維持する。
- **`okf_type` 属性の単純な破棄。** これは旧エクスポートが *type として
  書いていた綴り*なので、捨てると原表記が失われる。マイグレーションは
  §3.6 の通り**型に復元してから**属性を落とす。

## 5. 壊れるもの

破壊的変更を許容する方針(0017)の下でも、影響は語彙の綴りに限られる。

- **API / MCP の `type` 値**が入出力とも新表記になる。エイリアスを置かない
  (§3.5)ので、`--type tables` は**エラーにならず 0 件を返す** — 自由型
  `tables` として解釈されるためである。型でスイッチしていたクライアントと
  保存済みスクリプトは追随が必要。
- **型に空白が入る。** CLI とシェル補完はクォートが必要
  (`--type "Golden Query"`)。大小文字非依存の照合(§3.3)が実用上の
  緩和になる。
- **REST のクエリパラメータ**で型を渡す際に URL エンコードが要る。
  型はパスセグメントではないので、影響はクエリ文字列に限られる。
- **`okf_type` を読んでいたクライアント。** このキーは出力されなくなる。
- **空白を含む型を拒否していたクライアント側検証。**
- **エクスポートされたバンドルの `type` 行**は、複数形スラグで保存されて
  いた既存エントリでのみ変わる(`tables` → `BigQuery Table`)。もともと
  表示名で出力されていたので、**実際のバンドル出力に差分は出ない** —
  変わるのは内部表現と API 応答である。
