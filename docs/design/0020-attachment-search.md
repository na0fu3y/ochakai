# ochakai 設計ドキュメント 0020: 添付ファイルの検索

Status: Accepted(2026-07-20)
Date: 2026-07-20

## 1. 目的

設計ドキュメント 0013 は添付をファイル一般化しつつ、検索対象は
「従来どおりテキストのみ」と先送りした。しかし添付はエントリの根拠
そのもの — スキーマのシード、golden query の期待結果、仕様 PDF — で
あり、「seeds を付けたデータセットはどれ」「あの検証結果 CSV はどの
エントリに置いたか」が検索で見つからないのは、根拠を添付に置くほど
知識ベースが引けなくなるという逆転を生む。添付も検索の対象にする。

0013 が地ならしとして許可リストを揃えた gemini-embedding-2 は
2026-04-22 に Vertex AI で GA になっており(2026-07-20 実機確認)、
画像・PDF の埋め込みも本ドキュメントで敷く。

変えない原則が二つある。第一に、**添付は独立して返らない**(0008/
0013)。検索がヒットするのは所有エントリであり、添付はその中の
メタデータとして現れる。第二に、**ochakai は添付を解釈しない**
(OCR・キャプション・パースをしない — service/attachment.go の原則)。
埋め込みは決定的なエンコーダであり解釈ではない(embed パッケージの
原則)ので、この線は越えない。

## 2. 決定

### 2.1 レキシカル: ファイル名がハイスタックに入る

`attachment.name` を trigram 類似度と substring 一致の対象文字列に
加える。`seeds` や `q3-report` のようなファイル名の断片で所有エントリ
が当たる。埋め込み未設定(デフォルト構成)のインスタンスにもそのまま
効き、バックフィル不要で既存の全添付に即座に働く。

### 2.2 ベクトル: attachment_embedding テーブル

- `attachment_embedding (knowledge_id, name) PK, model, embedding, updated_at`。
  `knowledge_embedding` と同じく、埋め込みプロバイダが構成されたとき
  だけ `migrateEmbedding` が作る。
- **attach 時に埋め込む**。エントリ埋め込みと同じく、埋め込みの失敗は
  書き込みを失敗させない — ログに残して trigram 検索可能のまま進む。
  - `text/plain`: ファイル名+本文(エントリ body と同じ 4000 バイト
    切り詰め)をテキスト埋め込みに通す。どの埋め込みモデルでも働く。
  - `image/*`・`application/pdf`: 埋め込みモデルがファイル入力を
    取れる場合(§2.3)はバイト列を埋め込む。取れないモデルでは
    静かにスキップし、ファイル名レキシカル(2.1)だけで見つかる。
- **置き換え・detach で消える**: `PutAttachment` / `DeleteAttachment`
  のトランザクションが同名の埋め込み行を落とす(テーブルは条件付き
  存在なので savepoint で守る — SoftDelete の knowledge_embedding と
  同じ手筋)。置き換え後の再埋め込みは attach 経路が行う。
- **エントリの soft delete では消さない**。knowledge_embedding は
  復活(同 id への Create)が再埋め込みするから消せるが、添付は復活
  経路で再埋め込みされない。検索側の JOIN が `deleted_at IS NULL` で
  守るので、行を残しても削除済みエントリが浮上することはなく、復活
  すれば添付の検索も戻る。
- **検索**: クエリベクトルで attachment_embedding を引き、エントリ
  ごとに最良の 1 添付(`DISTINCT ON`)を取ってエントリのヒットに
  写像し、RRF フュージョンの第 3 のリストとして足す。本文でも添付
  でも当たるエントリは両リストから寄与を得て上がる — 根拠ごと当たる
  エントリが最初に来るのは意図どおり。

### 2.3 gemini-embedding-2 の embedContent 方言

gemini-embedding-2(GA 2026-04-22)は従来モデルと呼び方が違う。
2026-07-20 に ochakai-example で実機確認した事実:

- メソッドは `v1 :embedContent`(`:predict` は 404)。リクエストは
  `{"content":{"parts":[…]},"outputDimensionality":N}`、レスポンスは
  `{"embedding":{"values":[…]}}`。ファイルは
  `{"inline_data":{"mime_type":…,"data":<base64>}}` パートで渡し、
  テキストパートとの混在(ファイル名+バイト列)も取れる。
- `task_type` フィールドは無く、タスク指示をプロンプトに畳み込む:
  クエリは `task: search result | query: {content}`、ドキュメントは
  `title: {title} | text: {content}`(タイトル無しは `title: none`)。
- ロケーションは `global`・`us`・`eu` のみ(リージョン指定
  `us-central1` は 404)。`us`/`eu` のホストは素の
  `aiplatform.googleapis.com`。
- 上限: 全モダリティ共有で 8,192 トークン(超過は暗黙切り詰め)、
  PDF は 1 ファイル 6 ページまで、画像は 1 リクエスト 6 枚まで。
  次元は 128–3072 で `outputDimensionality` 指定、既定次元以外は
  L2 正規化済み。

Vertex ドライバはモデル名 `gemini-embedding-2*` でこの方言に切り替え、
ファイル埋め込み能力は `FileEmbedder` インターフェイスとして公開する。
従来モデル(`gemini-embedding-001` 等)は従来どおり `:predict` +
`task_type` で、ファイル埋め込みは非対応を返す。

**モデルの既定値は変えない。** ベクトルはクエリと同じ空間になければ
比較できないから、image/PDF を検索に載せるには knowledge base 全体を
gemini-embedding-2 で運用する(`OCHAKAI_VERTEX_MODEL=gemini-embedding-2`
と `OCHAKAI_VERTEX_LOCATION=global` 等)。既定を切り替えると既存
デプロイの保存済みベクトルが全て別空間の遺物になる(再インデックス
機構は無く、埋め込みは書き込み時のみ)ので、移行は運用者の明示的な
判断に委ねる。

### 2.4 既存添付のバックフィルはしない

エントリ埋め込みと同じ姿勢(埋め込みは書き込み時のみ。semantic
search を後から有効化した場合も、埋め込まれるのは以後の書き込み
だけ — deploy/cloudrun/README §4)。既存の添付は attach し直せば
載る。レキシカルのファイル名検索(2.1)は即座に全添付に効くので、
「見つからない」ことはない。

## 3. やらないこと

- **PDF のテキスト抽出・OCR を ochakai 自身が行うこと。** 解釈は
  クライアントエージェントの仕事(0008 以来の原則)。ファイルを
  読んで本文に書き戻せば、それは本文として検索に載る。
  gemini-embedding-2 の `document_ocr` オプションも使わない —
  既定で十分であり、オプション面を増やさない。
- **ヒットへの「どの添付が当たったか」の帰属表示。** 既存の検索も
  レキシカル/ベクトルどちらで当たったかを出していない。スコアの
  出自は今までどおり非公開の実装詳細に留める。
- **text/plain 本文のレキシカル検索(DB へのテキスト複製)。**
  バイト列は GCS のみ(0013)。Postgres に本文コピーを持ち込む
  二重管理はしない — ベクトル埋め込み(2.2)が意味検索を、
  ファイル名(2.1)が文字面検索を受け持つ。
- **モデル切り替え時の再インデックス機構。** 埋め込みは書き込み時
  のみという既存姿勢(2.4)の帰結。モデルを切り替えた運用者は、
  エントリ・添付を書き直すことで新空間に載せる。
- **許可リストの拡張。** gemini-embedding-2 は BMP/HEIC/音声/動画も
  取れるが、許可リストは「Claude が読めて Gemini が埋め込める」
  交わり(0013)のまま。読めないものは貯めない。
