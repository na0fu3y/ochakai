# ochakai 設計ドキュメント 0019: v0.10.0 リリース前の整合調整 — ID 文字種の OKF 同等化・インポートのスキップ拡張・compile のモデル解決

Status: Accepted(2026-07-19。0016/0017/0018 のリリース前レビューでの決定)
Date: 2026-07-19

## 1. 目的

0016〜0018 の実装レビューで見つかった、設計判断を要する 3 点を確定する。
いずれも 0.10.0 に同乗する(未リリースの変更への追記であり、刻む理由がない)。

1. **ID セグメントの文字種を OKF 同等まで緩和する**(0017 §4.1 の改訂)。
2. **バンドルインポートはサーバーが拒否した文書もスキップしてレポートする**
   (0018 の書き込み時検証との整合)。
3. **compile のモデル暗黙解決を `metrics/<name>` 固定パスから
   models エントリ自身の走査に変える**(0018 §4.3 の改訂)。

## 2. ID セグメントの文字種: パス安全性だけを制約にする

0017 は「concept ID = パス」を採ったが、セグメントの文字種は 0005 §3.1 の
スラグ規則(先頭英数字 + `[a-zA-Z0-9._-]`)のまま残していた。一方 OKF SPEC は
ファイル名の文字種を一切規定しない — 予約するのは `index.md` / `log.md` の
2 ファイル名だけである。スラグ規則は次を弾いてしまう:

- 先頭 `_` のセグメント。BigQuery では `_` 始まりのデータセット
  (隠しデータセット)・テーブル名が合法で、外来バンドルの
  `tables/_intraday.md` がスキップされていた。
- 非 ASCII。日本語の用語集を `用語/売上` に置けない。
- 空白などの一般的なファイル名文字。

### 決定

セグメントの制約を**パス安全性のみ**にする: 非空の正しい UTF-8、
128 バイト以下、制御文字なし、**先頭 `.` 禁止**(`.` / `..` / 隠しファイルを
一括で排除 — 隠しパスはインポートも「ナレッジではない」としてスキップする
既存規則と同じ線)。ID 全体は従来どおり 512 バイト以下、末尾セグメントの
`index` / `log` 予約も変更しない。

**type は従来のスラグ規則のまま**とする。type はパスではなく語彙であり
(0017 §3)、フィルタ・frontmatter・ツール引数で綴られる。自由 type の
スラグ化(0005 §2)も従来どおり機能する。

なお緩和はバリデーションの緩和であり、マイグレーションは不要
(既存 ID はすべて新規則の部分集合)。

## 3. インポート: サーバーの 400 は「その文書のスキップ」

0017 §4.6 のインポート規則は「エントリになれないファイルはスキップして
レポートする」— だが CLI 実装はサーバーが 400 で拒否した文書(0018 の
書き込み時検証に落ちる models エントリなど)で**バンドル全体を中断**
していた。外来バンドルが自由語彙として `models` を使っていた場合、
途中まで取り込んだ状態で止まる。

### 決定

`ochakai import` はサーバーの 400(入力起因の拒否)を**その 1 文書の
スキップ**として扱い、レポートに載せて続行する。スキップされたエントリに
帰属する添付も併せてスキップする。認証・ネットワーク・5xx は従来どおり
中断する — パイプラインの故障と 1 文書への判定は別物である。

## 4. compile のモデル解決: モデル自身が正

0018 §4.3 は model 省略時の解決を「`metrics/<第1メトリクス名>` エントリの
attrs.model」とした。これは 0017 が廃止したはずの固定パス規約をサーバーに
1 箇所残し、メトリクスエントリをドメイン先頭レイアウトに置いた利用者は
暗黙解決を失っていた。

### 決定

model 省略時、サーバーは **models エントリ自身を走査**する:
`attrs.spec.metrics[].name` に第 1 メトリクス名を定義する live な models
エントリが**ちょうど 1 つ**ならそれを使う。0 なら「モデルを作るか model を
渡せ」、複数なら候補の id を列挙して「model で選べ」という明確な
compile エラー(422)を返す。モデルはメトリクスの真実の源(0018 §4.4 —
ずれはモデル側を正としてエラーになる)であり、解決もモデル側で行うのが
一貫する。パス規約への依存はゼロになる。

- メトリクスエントリの `attrs.model` は**検索・ナビゲーション・usage 帰属の
  ための規約として存続**する(compile usage は、解決したモデルを
  attrs.model で指す metrics エントリのうち、id 末尾セグメントが要求
  メトリクス名に一致するものへ記録する)。存在しないエントリへの usage
  記録(ghost 行)はこれで消える。
- `CompileRequest.model`(明示指定)の意味は不変。曖昧さは利用者が
  明示指定で解く — サーバーは推測しない(0018 §3 と同じ線)。

## 5. やらないこと

- **type 文字種の緩和。** §2 のとおり type は語彙。
- **インポートの 4xx 全般のスキップ化。** 404/409 等は既存の分岐
  (create → conflict → update)の一部であり、変更しない。
- **メトリクス名の大域一意制約。** 複数モデルが同名メトリクスを定義するのは
  合法のまま。曖昧なら明示、が唯一の規則。

## 6. 壊れるもの

- **旧スラグ規則を前提にした ID バリデーション**: 先頭 `_`・非 ASCII・
  空白入りの ID が新たに合法になる(受理が広がる方向のみ。既存 ID は不変)。
- **compile の暗黙解決の意味**: `metrics/<name>` エントリの attrs.model では
  なく models エントリの走査になる。移行済みデータ(0012)ではどちらも同じ
  モデルに解決するため、実挙動の差は「メトリクスエントリ未作成でも
  モデルさえあれば compile できる」「同名メトリクスを持つモデルが複数あると
  明示指定が要る」の 2 点。
- **import の部分成功**: 従来は 400 で中断していたバンドルが、スキップ付きで
  最後まで走る。終了コードは成功(スキップはレポートで見える)。
