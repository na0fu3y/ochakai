# ochakai 設計ドキュメント 0022: ファイル名が名前 — title の任意化・ID の検索対象化・NFC 正規化

Status: Accepted(2026-07-20)
Date: 2026-07-20

## 1. 目的

0019 の文字種緩和で `insights/サンプル` は既に合法な ID である。にもかかわらず、
日本語で書く利用者の自然な形 —「ディレクトリとファイル名だけ決めて、名前は
ファイル名に書く」— は今日 3 つの理由で成立しない。

1. **title が必須。** `{id: insights/サンプル}` と置いても、サーバーは別途
   title を要求する。名前を 2 回書かされ、ずれた瞬間にどちらが正か曖昧になる。
   ASCII スラグ時代の名残であり(`{id: insights/sample, title: サンプル}` は
   スラグと表示名の分離が必然だった)、ファイル名に名前を書ける今、必然性が
   消えている。
2. **検索が ID を見ない。** レキシカル検索・エンベディングの対象は
   title + description + tags + body(+ 添付名)で、名前をファイル名側に
   置いたエントリは名前で引けない。
3. **ID の Unicode 正規化が未規定。** ID はバイト比較の主キーだが、macOS の
   ファイルシステムはパス名を NFD 分解して返す。バンドルを macOS で展開して
   再 tar すると、見た目同一の `サンプル`(NFC)と `サンプル`(NFD)が
   **別エントリ**として静かに二重化する。ASCII スラグの世界では起こり得な
   かった問題で、0019 が非 ASCII を受けた時点で潜在化していた。

## 2. 世界観: パスが名前を兼ねる

0017 は「パスが住所、タイプは属性」とした。本ドキュメントはその続きである —
**住所の末尾(ファイル名)は、多くの場合そのまま名前である**。特に日本語では、
スラグ化した別名を作る動機がない。

- title は「ファイル名で足りないときのオーバーライド」に降格する。
  ファイル名 42 文字(128 バイト)に収まらない説明的な題、`/` を含む題、
  外来バンドルが持ち込む題 — そのときだけ書く。5 タイプ推奨と同じ
  「推奨はするが、強制しない」の適用である。
- OKF SPEC との整合はむしろ改善する。SPEC が必須とする frontmatter キーは
  `type` だけであり、title は推奨キーにすぎない。title 無し文書を 400 で
  弾く現状のほうが SPEC より狭かった(0017 の「type 無しはスキップ」と
  違い、title 無しは concept として完全に合法)。
- NFC 正規化は「同じ見た目のパスは同じエントリ」という、ファイルシステムの
  利用者が当然に期待する等価性の回復である。W3C の文字モデルも交換形式に
  NFC を推奨している。

## 3. 決定

### 3.1 title は任意 — 表示は末尾セグメントにフォールバック

- サーバーの title 必須検証を撤廃する。DB は `NOT NULL` のまま空文字を許す
  (マイグレーション不要)。
- **表示名 = title があれば title、なければ ID の末尾セグメント**
  (`domain.DisplayTitle`)。Web UI(検索カード・ツリー・詳細・ドラフト
  キュー)、CLI(search / browse / backlinks / context)、エクスポートの
  index.md(既にこの規則で実装済み)のすべてが同じ規則に従う。
- **エクスポート**: title が空なら frontmatter に `title` を出力しない
  (omitempty)。空のときだけ省略する — 「末尾セグメントと同一なら省略」の
  ような積極的正規化は**やらない**。外来バンドルの byte 同一往復
  (0005 §3.3)を壊すためである。
- **インポート**: title 無し文書はスキップされず取り込まれる(title 空の
  エントリになる)。往復は安定する: title 無しで入った文書は title 無しで
  出て行く。
- domain.Knowledge の JSON も `title,omitempty` になり、空 title は
  ワイヤ上で欠落する。

### 3.2 ID は検索対象

- **レキシカル検索**: similarity のヘイスタックと ILIKE の部分一致フロアに
  `k.id` を加える。`用語/売上` は「売上」で引ける。
- **エンベディング**: embeddingText の先頭に ID を加える。パスは
  ドメイン階層(0017 §4.8「一緒に読ませたいものを同じディレクトリに」)を
  含むので、title があるエントリでも文脈が増える。
- 既存エントリのベクトルは旧テキストのままである(0020 と同じ性質)。
  再保存で更新される。

### 3.3 ID・添付名・検索クエリは NFC に正規化して保存・照合する

- `domain.Normalize`(Unicode NFC)を導入し、**バイト比較される文字列が
  システムに入るすべての境界**で適用する:
  - service の ID を受ける全入口(get / create / update / delete / move /
    revisions / backlinks / usage / report_outcome / browse の prefix)
  - create / update ペイロードの links の target
  - 添付の ID と名前(attach / get / detach)
  - バンドルインポートのパス(concept のパス、添付のパス、body リンクの
    解決結果 — macOS 由来の NFD がここから入る)
  - 検索クエリ(NFC で保存された本文と ILIKE で照合するため)
- 正規化は入口のみで行い、DB 内の照合は従来どおり素のバイト比較のまま。
- **既存データ**: マイグレーション 0013 で全 ID 列・添付名・links の
  target・リビジョンスナップショットの id を `normalize(…, NFC)` に
  書き換える。実データはほぼ全て ASCII で no-op だが、0019(v0.10.0)以降に
  NFD の ID が書かれ得た期間があるため、不変条件は移行で真にする。
  正規化衝突(NFC 化で同一になる 2 エントリ)は理論上の存在で、移行は
  素直に失敗して人間に知らせる。
- body・title・description は正規化**しない**。内容は書かれたまま —
  正規化するのは鍵として比較される文字列だけである。

### 3.4 「フォルダ + 名前 → id」はクライアントで組む

サーバーの住所は id 一本のままで、導出は入れない(0017 / 0019 の
「推測しない」の線。検討済み代替案: サーバーが directory + title から id を
導出する案は、正規化・衝突・リネームの意味論をサーバーに持ち込むので却下)。
入力体験はクライアントの仕事である:

- **Web UI の新規作成フォーム**: ID の 1 フィールドを **Folder + File
  name の 2 フィールド**にする(id = `folder/name`、folder 空ならルート
  直下)。ツリーの各ディレクトリの ＋ からは folder が事前入力される。
  Title は任意入力になり、「省略時はファイル名が名前」とヒントで示す。
  編集時は従来どおり id 固定表示。
- **CLI**: `ochakai create <id>` は既に「パス引数 + ドキュメント」であり、
  title 任意化により `printf '---\ntype: insights\n---\n本文' |
  ochakai create insights/サンプル` の最小形が成立する。usage の文言のみ
  更新する。

## 4. やらないこと

- **title 列の廃止。** ファイル名に収まらない題・外来バンドルの題の置き場
  として存続する。表示名だけ直したいときに move(参照書き換え・リビジョン)
  を強いるのも過剰。
- **サーバー側の id 導出**(§3.4)。
- **「title == 末尾セグメントなら export で省略」の正規化**(§3.1)。
- **ID 以外の NFC 正規化**(§3.3)。内容は書かれたまま。
- **NFD で検索したときの互換照合。** クエリは NFC 正規化されるので実用上
  困らない。DB 側に正規化付き照合(COLLATE や関数インデックス)は足さない。

## 5. 壊れるもの

破壊的変更を許容する方針(0017)の下でも、本改訂はほぼ「受理が広がる方向」
であり、壊れるものは少ない。

- **openapi / MCP スキーマの title**: required から外れる。title 非空を
  仮定していたクライアント表示は、空(または欠落)title に備えるか、
  末尾セグメントへのフォールバックを実装する必要がある。
- **JSON ワイヤ形**: 空 title は `"title": ""` ではなく欠落になる
  (`omitempty`)。
- **NFD の ID・添付名・リンク target**: 書き込み時に NFC へ正規化される
  (見た目は同一)。NFD の綴りを保持して往復することはできなくなる。
  マイグレーション 0013 は保存済みの NFD をまとめて NFC に書き換える。
- **エクスポートの title 行**: title 空のエントリで `title:` 行が消える
  (今までは空 title が存在し得なかったので、実際の diff は出ない)。
