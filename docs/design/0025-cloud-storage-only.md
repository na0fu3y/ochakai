# ochakai 設計ドキュメント 0025: Cloud Storage 一本化

Status: Proposed(2026-07-24)
Date: 2026-07-24

## 1. 目的

ochakai は今、状態を **Cloud SQL PostgreSQL** に置く(0003)。本ドキュメントは
その代替として、**永続層を Google Cloud Storage(GCS)一本にする**構成を提案し、
実験でその成立性を検証する。

動機は 0003 と同じ「簡略化」の延長線上にある。0003 は認証を cloudrun-iam に
一本化し、静的トークンという概念ごと捨てた。本改訂はその姿勢を**永続層にも
適用する** — 常時稼働の VM(Cloud SQL)を持たず、シークレットも VPC も増やさず、
依存を「オブジェクトストア一つ」に畳む。

これは 0003 を supersede しない。0003 の核(GCP 前提・IAM 認証・シークレット
ゼロ・データ可搬性)は**むしろ強まる**。置き換わるのは「状態の置き場所が
Cloud SQL である」という機構だけである。

### 1.1 出発点となった観察

ochakai の DB が本当にやっている仕事を棚卸しすると、以下に尽きる。

- 数千〜数万件の JSON ドキュメント(エントリ)を、複数インスタンスから
  **原子的に**、**履歴つきで**永続化する
- その上で日本語のレキシカル検索とベクトル検索を提供する

そして検索は、コード自身が既に「ANN 索引は要らない、全走査で十分」と認めて
いる(`internal/store/migrate.go` の `migrateEmbedding` コメント)。数万件 ×
768 次元 = 数十 MB は、プロセス内で総当りして数ミリ秒で終わる規模である。

つまり **DB は検索エンジンである必要がなく、「数十 MB の JSON を安全に永続化
する」ことが本質**である。これを認めると、Cloud SQL でなければならない理由が
消える。

## 2. 世界観: オブジェクトが正本、スナップショットは索引

0017「パスが住所」、0022「ファイル名が名前」、0024「本文が関係」の系列に、
本改訂は 4 つ目を足す — **オブジェクトが状態である**。

```
                      ┌─────────────────────────────┐
   書き込み(CAS)  →  │  GCS: 1 エントリ = 1 オブジェクト │  ← 正本(source of truth)
                      │  entries/<id>.json (versioned)  │    バージョニング = 履歴
                      └───────────────┬─────────────────┘
                                      │ フルビルド(定期・オフライン)
                                      ▼
                      ┌─────────────────────────────┐
                      │  GCS: snapshot.sqlite + vec.f32 │  ← 索引(cache)
                      └───────────────┬─────────────────┘
                                      │ 起動時 1 GET → ephemeral disk
                                      ▼
   読み取り        ←  各 Cloud Run インスタンス(読み取り専用でマウント)
```

- **正本**: エントリごとに 1 つの GCS オブジェクト。書き込みは
  `ifGenerationMatch` によるオブジェクト単位の CAS。オブジェクト
  バージョニングが `knowledge_revision` を置き換える。
- **索引**: 全エントリを畳んだ読み取り専用の SQLite ファイル(レキシカル)と
  フラットな float32 ファイル(ベクトル)。ビルダーが定期的に焼く**キャッシュ**
  であって正本ではない。
- **配信**: 各インスタンスは起動時にスナップショットを 1 回 GET し、Cloud Run の
  ephemeral disk(RAM を消費しない)に置いて読み取り専用で使う。

## 3. 検証: プロセス内検索は pg_trgm を再現できるか

**前提**: 検索を DB から出してプロセス内(またはスナップショット)に移すには、
`internal/store/store.go` の `SearchLexical` が使う `pg_trgm.similarity()` の
ランキングを、外で忠実に再現できねばならない。

**方法**: ochakai 自身の設計ドキュメント 24 本を見出しで分割した 208 エントリ、
本文由来の 142 クエリで、`SearchLexical` の採点式(similarity + ILIKE の +0.3 +
verified の +0.05)を全 29,536 ペアで両実装採点し突き合わせた。

**結果**:

| 指標 | 値 |
|---|---|
| top-10 順位の完全一致 | 141/142(99.3%) |
| top-10 集合の一致 | 141/142(99.3%) |
| 平均 Jaccard@10 | 0.9987 |
| スコア最大差 \|Δ\| | 0.0027(10位の同点入れ替わり 1 件のみ) |

唯一の差の原因も特定した。pg_trgm はマルチバイトのトライグラムを **CRC32 で
3 バイトに圧縮**して格納する(208 エントリ中 5 件で衝突、失トライグラム 6 個)。
差は常に一方向で、**プロセス内実装のほうが真のトライグラム集合に忠実、
PostgreSQL 側が近似**。品質は落ちるどころか、わずかに上がる方向である。

**結論**: pg_trgm はプロセス内でできないことを何もしていない。検索は永続層
選択の制約ではなくなった。

## 4. 検証: 索引の置き場所 — メモリか、スナップショットか

プロセス内索引には 2 つの形がある。実測(10 万エントリまで)で比較した。

### 4.1 メモリ索引(却下)

| エントリ | 本文 | レキシカル | ベクトル | 合計/インスタンス | 起動時構築 |
|---|---|---|---|---|---|
| 5,000 | 8.5 MB | 2.7 MB | 15.5 MB | 26.6 MB | 0.5s |
| 20,000 | 34 MB | 10.7 MB | 61.9 MB | 106.6 MB | 2.1s |
| 100,000 | 170 MB | 53.7 MB | 309.6 MB | 533.3 MB | 10.8s |

2 つの壁がある。(i) 索引が**インスタンス数だけ複製**される(50 インスタンス ×
533 MB)。(ii) コールドスタートのたびに**索引を再構築**する(10 万件で 10.8 秒、
起動プローブが通らない)。

### 4.2 事前ビルドの SQLite スナップショット(採用)

索引を「起動時に組み立てるもの」から「あらかじめ組み立ててあるファイル」に
変える。10 万エントリで実測:

| | メモリ索引 | SQLite スナップショット |
|---|---|---|
| 起動時の索引構築 | 10.8 秒 | **0 秒**(構築済みで届く) |
| クエリ時の実メモリ | 533 MB | **2.4 MB**(pure-Go ドライバ、ヒープ) |
| レキシカル検索 | 22.5 ms | 78–195 ms(bm25 ランク込み) |
| ファイルサイズ | — | 831 MB(本文の約 5 倍、trigram 索引ぶん) |
| オフライン構築 | — | 39.5 秒(1 回) |

831 MB のファイルに対するクエリのヒープが **2.4 MB**。SQLite は索引を常駐
させず、必要なページだけをディスクから引く。Cloud Run の ephemeral disk は
メモリを消費しないので、512 MB インスタンスに載る。

しかもこれは**読み取り専用・不変・バージョン付きの成果物**なので、SQLite を
Cloud Run で使う際の典型的な難点(GCS FUSE のロック非対応、複数インスタンスの
破損)が**全て消える。誰も書き込まないから**。書き込みは個別 GCS オブジェクト
(正本)へ、ビルダーが定期的に SQLite を焼く、という二層で分離される。

### 4.3 ベクトルはどちらの DB にも入れない

ベクトルはフラットな float32 ファイルに置き、Go でストリーム走査する。10 万件
・768 次元(293 MB)を **150 ms・ヒープ 3 MB** で全走査(実測)。`migrate.go` が
「数千行なら全走査で十分」と認めている全走査を、そのまま外へ移すだけなので
結果はビット単位で同一。

## 5. 検証: SQLite か DuckDB か

配信層の索引を SQLite にするか DuckDB にするかは、**層で答えが割れる**。実測と
一次情報で確認した。

### 5.1 配信層 = SQLite(pure-Go modernc)

決め手は 3 つ、いずれも検証済み。

1. **trigram トークナイザの有無**。DuckDB の FTS は Porter ステマー(英語)
   ベースで **trigram を持たず、日本語は外部拡張が要る**(duckdb/duckdb
   discussions #16071、duckdb-vaporetto)。ochakai の唯一の必須要件(日本語
   部分一致)を DuckDB は素で満たせない。対して **pure-Go の
   `modernc.org/sqlite`(cgo なし)は FTS5 trigram + bm25 を積んでいる**
   ことを実測で確認した(version 3.53.3、日本語 3 文字クエリでヒット)。
2. **cgo の有無**。DuckDB の Go ドライバは cgo + 約 30 MB のネイティブ静的
   ライブラリで、`CGO_ENABLED=0` にできず richer base image が要る。modernc
   SQLite は pure-Go・cgo ゼロ・distroless/scratch に載る。cgo を持たない
   現コードベース(pgx)の姿勢に一致する。
3. **エンジンの向き**。DuckDB は列指向 OLAP で真価は大規模集計スキャン。
   ochakai の配信(ID 直引き・FTS・小フットプリント)は DuckDB の核心能力を
   一度も使わない。SQLite の実測は点引き 3.6 ms・ヒープ 2.4 MB。

### 5.2 ビルド層 = 素の Go ビルダー

DuckDB の強み(GCS 上の Parquet 直読み、一括変換)はビルダー側の話だが、
ビルダーは既存の Go で足り、変更分だけを処理する増分フルビルド(§6)は分析
クエリではない。DuckDB は将来の分析用の任意オプションに留め、**サーバには
載せない**。

## 6. 書き込みプロトコルとフルビルド

### 6.1 GCS の意味論(前提)

以下の GCS の**公表された**性質だけに依存する
(cloud.google.com/storage/docs/consistency)。

- オブジェクト一覧(LIST)は強整合
- read-after-write は強整合
- `ifGenerationMatch` がオブジェクト単位の compare-and-swap を与える
- `objects.list` は各オブジェクトの `generation` と `crc32c` を返す
  (GET せずに「変わったか」を判定できる)

### 6.2 書き込み: 3 つのルール

**ルール 1 — 書き込みは CAS。** `Update` は「GET(gen N)→ 変更 →
PUT ifGenerationMatch=N」。競合すれば 412 で弾かれ、再読込してリトライする。
これは**現状の PostgreSQL より強い**。現 `internal/store/store.go` の `Update` は
`WHERE id=$1` の無条件 UPDATE で、同時編集が黙って last-write-wins する
(§10 の前提条件)。GCS 移行は競合検出を獲得する。

**ルール 2 — 指される側を先に書く。** リビジョンオブジェクト(エントリが指す先)を、
それを指すエントリより先に create-only PUT する。どのクラッシュ地点でも、
孤児リビジョン(ゴミ、GC 可能)は残りうるが**宙吊り参照は生じない**。これで
entry + revision の 2 オブジェクト原子性を、トランザクションなしで達成する。

**ルール 3 — スナップショットはキャッシュ、網羅はフルビルドの LIST が担保。**
completeness の根拠は「フルビルドの LIST が強整合」の一点。ビルド K の LIST
時点で存在した書き込みは必ず K に入る。ビルド中に生まれたものは K が取りこぼす
が、K+1 の LIST が必ず拾う。増分マーカーを持たない限り、永久取りこぼしは
**構造的に不可能**。

### 6.3 モデル検査

上記を GCS の意味論だけでモデル化し、ランダム並行スケジュールに不変条件を
突きつけた。**20 万試行・90 万書き込み**、圧力として **64 万回の CAS 衝突**・
**12 万回のクラッシュ注入**・**15.7 万回の torn build**(LIST と GET の間で書き込みが
進行)を注入。

| 不変条件 | 結果 |
|---|---|
| INV1 ロスト更新なし(同一 rev の二重 ack なし) | 全試行で成立 |
| INV2 宙吊り参照なし(全原子ステップ後に検査) | 全試行で成立 |
| INV3 最終的網羅性(ack した書き込みは後続ビルドに必ず出る) | 全試行で成立 |

torn build は実際に **4.9% が瞬間的に不完全**になったが、その取りこぼし
(のべ 19,052 件)は**次のフルビルドが 1 件残らず回復**した。主張は「取りこぼし
ゼロ」ではなく「**永久取りこぼしゼロ**」であり、これは受容する鮮度窓そのもの。

### 6.4 フルビルドの成果物別コスト構造

「全件 LIST して変更分を処理」は正しいが、「変更分」= **新規 + 更新 + 削除**で
あり、処理コストは成果物ごとに 3 層に分かれる。

| 成果物 | キャッシュ鍵 | 毎回フル再構築か | 理由 |
|---|---|---|---|
| **埋め込み** | 埋め込み対象テキストの内容ハッシュ | **絶対に不可** | Vertex は課金 + レート制限。変更分だけ再計算 |
| 添付の埋め込み | blob の sha256 | 不可 | 添付は content-addressed(0004/0011) |
| **検索索引(trigram)** | — | **毎回フルで可** | 純 CPU、10 万件 40 秒。増分不要 |
| links(順方向) | body の内容ハッシュ | パースだけキャッシュ | 0024 で body から導出 |
| **link_from(逆リンク)** | — | フルビルドの産物 | 全 body の畳み込み。全件必須 |

変更判定はスナップショットが記録した `id → generation` を LIST 結果と突き合わせて
下す(キューも時計も使わない = §6.2 ルール 3 の completeness を保つ)。**埋め込み
コストはコーパスサイズではなく変更率に比例**する — 10 万件・変更 200/日なら
1 日 200 回の埋め込みで済む。逆リンクは全 body を持つフルビルドが 1 パスで反転
する、**フルビルド方式の追い風**となる成果物である。

## 7. 鮮度モデル

- **fetch by id・ブラウズ**: GCS を直接読むので**強整合**(read-your-writes)。
  「保存 → 詳細を開く」は正しく動く。
- **検索**: スナップショットの再ビルド間隔ぶん遅れる。「保存 → 検索で探す」だけが
  遅延する。鮮度窓の下限は「フルビルド時間(10 万件で約 40 秒)+ アップロード +
  各インスタンスのダウンロード + ビルド間隔」。

これは「検索索引が数分遅れる wiki」であり、製品として説明可能な範囲に収まる。
運用上の必須条件として、**スナップショット世代の鮮度をメトリクスに出す**
(障害時に検索結果が静かに古くなる、を検知するため)。

## 8. コスト

`asia-northeast1`・Standard、5,000 エントリ、書き込み 200/日、読み 2,000/日。
公表標準レートを前提とした計算(実請求では未検証)。

| 設計 | 月額 | 備考 |
|---|---|---|
| 素朴(1 エントリ = 1 オブジェクト、起動時に全 GET) | $6〜123 | コールドスタート回数に比例。**却下** |
| スナップショット(起動時 1 GET) | **$0.06〜0.2** | コールドスタート回数にほぼ非依存 |
| 保存(20 世代バージョニング込み) | $0.005 | 無視できる |
| (参考)現行 Cloud SQL f1-micro | $9.4 | |

残コストのほぼ全額が**書き込み**(スナップショット再構築の Class A)で、読み取りは
メモリ/ディスク索引が答えるため GCS 操作が出ない。落とし穴: `objects.list` は
Class A(Class B の 12.5 倍)なので、鮮度検知に定期 LIST を使うと高くつく。
変更検知は LIST ではなく**小さなマニフェスト(id→generation)を 1 GET**で行う。

**コスト単独では移行の動機にならない**($9.4 は工数に見合わない)。動機は
「アイドル $0・依存を 1 つに・可搬性が runtime の事実になる」であって、コストは
副産物である。

## 9. スケール上限(正直な棚卸し)

限界は「人数」ではなく「**エントリ数**」で決まる。

| 軸 | 5,000 | 20,000 | 100,000 |
|---|---|---|---|
| SQLite スナップショット | 快適 | 快適 | 40 秒ビルド・831 MB |
| フルビルド時間(鮮度窓の下限) | 数秒 | 約 10 秒 | 約 40 秒 |
| ベクトル全走査/クエリ | 5 ms | 19 ms | 93 ms |

- スナップショット方式は §4.1 の 2 つの壁(RAM × インスタンス数、コールドスタート
  構築)を消す。残る本質的コストはフルビルド時間による鮮度窓の下限のみ。
- **10 万件前後でベクトル全走査が 93 ms/クエリ**に達し、ANN 索引が必要になる。
  ただしこれは pgvector 側も同設定なので**永続層選択とは独立**の課題。
- 1000 人規模では usage が最初に壊れる(§10)。これも永続層と独立。

この設計は 0001 の「規模より信頼密度」「小〜中規模チーム」という自己規定と
正確に一致する。**全社 1000 人カタログに売るなら Cloud SQL / AlloyDB を維持
すべき**で、その規模で GCS 一本を成立させる作業は ochakai 本体より大きい分散
システムの構築になる。

## 10. トランザクション境界の外に出すもの

| 対象 | 境界 |
|---|---|
| 単一エントリ更新 | 境界内。CAS で原子的・競合検出あり |
| entry + revision | 境界内。§6.2 ルール 2 |
| 検索索引(スナップショット) | 境界外・結果整合。永久取りこぼしなし |
| **usage カウンタ** | **意図的に境界外**。at-least-once、クラッシュ時ロス許容 |

usage が唯一の妥協点だが、これは新たな妥協ではない。現
`internal/store/usage.go` は既にコメントで「usage recording must never fail a
read」と書き、**読み取りパスで同期 UPSERT しつつ失敗を握り潰している**。GCS は
単一オブジェクトへの持続書き込みが約 1 回/秒に制限されるため、人気エントリの
カウンタをオブジェクトにすると 1000 人規模で超える。統計値が数件ずれるのを許すか、
正確さが要るなら Firestore の分散カウンタ / BigQuery 追記に逃がす。

## 11. 前提条件(どの道を選んでも必要な修正)

以下は移行の前提であると同時に、**現状のままでも直すべき**バグ/負債である。
Cloud SQL 上で先に入れれば、移行の切り分けになる。

1. **`Update` に楽観ロックがない**(`internal/store/store.go` の `Update` は
   `WHERE id=$1` の無条件 UPDATE)。人間とエージェントの共同編集が黙って
   last-write-wins する。GCS の `ifGenerationMatch` と同じ意味論
   (期待 generation / updated_at の事前条件)を先に PostgreSQL 上で導入する。
2. **usage の同期書き込み**。`RecordEvents` を読み取りパスから外し、インスタンス内
   バッファ + 定期フラッシュにする。どのバックエンドでも必要。
3. **ILIKE のワイルドカード未エスケープ**。`SearchLexical` の substring boost が
   `%` / `_` を含むクエリで全エントリに +0.3 を付け、順位を平坦化する。
   プロセス内実装に移せば自然に消えるが、現状でもバグ。

## 12. 検討した代替(否定せず並べた上での棄却)

- **Cloud SQL 継続(現状)**: 全要求を満たす唯一の既製品。不満はアイドル $0 に
  できない一点。**コストが動機なら移行しない**が正解。接続数は f1-micro で 6〜8
  インスタンスで枯渇し、100 人前に upsize が要る(GCS は接続上限がない)。
- **AlloyDB / AlloyDB Omni**: PG 互換で移行は楽だが最小構成でも月 $100 超。数万件に
  ScaNN と列指向は過剰。全社カタログ級になった時の逃げ道として保持。
- **Cloud Spanner**: 技術的には唯一の全部入り(日本語トークナイザ付き FTS +
  ベクトル + 強整合 + 運用ゼロ)。ただしフリーティアなし・月 $22〜。**検索品質が
  動機なら**唯一の候補だが、コスト動機とは真逆。
- **Spanner Omni**: その名の GA 製品は存在しない(emulator は非永続、PGAdapter は
  ワイヤ互換層)。検討対象外。
- **Bigtable**: 最小 1 ノード月 $50〜、クロス行トランザクションなし、検索なし。
  ochakai のアクセスパターンに一つも合わない。却下。
- **Firebase Realtime Database**: クエリが弱くベクトルなし。Firestore の下位互換。
- **Firestore**: GCS 一本とほぼ同じ崖(コールドスタートで N ドキュメント読むか)と
  同じ回避策(スナップショット)を持つ。分散カウンタで §10 が解け、接続上限も
  ない点は GCS より楽。**GCS でなく Firestore にする判断も合理的**で、両者の差は
  コストでは決まらない。本ドキュメントは「依存を 1 つに」を最優先して GCS を採る。
- **DuckDB**: §5。配信層では trigram 非対応 + cgo で不適。ビルド/分析の任意
  オプションに留める。
- **SQLite を書き込み DB として使う**: GCS FUSE は POSIX ロック非対応で破損しうる
  (Google 自身が本番非推奨)、複数インスタンス不可。読み取り専用スナップショット
  としてのみ採用(§4.2)。CLI 側のローカル検索用途にも別途最適。

## 13. 壊れるもの / 移行

- **`internal/store` の全面書き換え**。pgx 依存の `Store` が GCS クライアント +
  スナップショットリーダに替わる。`Migrate`(versioned SQL)は不要になり、代わりに
  スナップショットのスキーマバージョンとビルダーが要る。
- **ビルダーの新設**。GCS を LIST → 変更分を処理 → SQLite + float32 を焼いて GCS に
  置く常駐/定期ジョブ(Cloud Run Job か、書き込みを受けたインスタンスが債務化)。
- **既存データの移行**。Cloud SQL の各行を `entries/<id>.json` として書き出す
  一括エクスポート。履歴(`knowledge_revision`)はオブジェクトバージョン、または
  `revisions/<id>/<rev>.json` として書き出す。
- **ローカル開発**。docker compose の PostgreSQL が不要になり、GCS エミュレータ
  (fake-gcs-server)かローカルファイルシステムのアダプタに替わる。
- **0003 の環境変数**。`OCHAKAI_DATABASE_URL` / `OCHAKAI_DB_IAM_AUTH` が
  `OCHAKAI_GCS_BUCKET`(添付で既出)系に置き換わる。

破壊的変更が大きいため、リリースは独立した専用の列車に載せる。段階移行として
「Cloud SQL を正本のまま、スナップショットを GCS に焼いて検索だけ切り替える」
中間形も取りうる(§11-1/2 を入れた後)。

## 14. 未解決 / リスク

- **実 GCS の未検証項目**: スナップショットの実転送速度、`ifGenerationMatch` の
  412 挙動、create-only の正確な意味論、`objects.list` のページング料金。
  §6–8 はドキュメント上の意味論と公表レートを前提とした計算/モデルであり、
  統合テストでの実測が要る。
- **FTS5(bm25)の検索品質そのもの**は未測定。§3 は「pg_trgm スコアの再現」を
  示したが、スナップショットで bm25 を使うとランキングが変わる。**FTS5 trigram で
  候補を絞り、絞った候補にだけ §3 の pg_trgm 互換スコアを掛ける**ハイブリッドで
  忠実性を保つ想定だが、これは次の実験対象。
- **2 文字クエリ**: FTS5 trigram は 3 文字未満を引けない(実測)。日本語の 2 文字
  クエリ(添付・検索・認証)は最頻出。LIKE 全走査フォールバック(10 万件 0.28 秒)か、
  上記ハイブリッドの候補生成で吸収する。
- **ビルダーの単一障害点化**。ビルダーが止まると鮮度が凍る(読み取りと fetch by id は
  生きる)。鮮度メトリクスとアラートが必須。
