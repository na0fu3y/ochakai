# ochakai 設計ドキュメント 0021: ナレッジの move(パス変更)と Web UI の微調整

Status: Accepted(2026-07-20)
Date: 2026-07-20

## 1. 目的

ツリー常設サイドバー化(0014/0016/0017)後の Web UI 実運用で見えた摩擦を
解消する。

1. **エントリのパスを移動できない。** ID がアドレスである以上(0017)、
   「置き場所を変える」は日常の整理作業だが、これまでは
   「新 ID で作り直し → 参照を手で張り替え → 旧を削除」しかなく、
   リビジョン・使用実績・添付が旧 ID 側に取り残された。
2. **ディレクトリ配下への新規作成が遠い。** ＋ から白紙のエディタが開き、
   フル ID を手入力する必要があった。
3. **表示の雑音。** サイドバーの各行に ID・タイトル・ステータスバッジが
   全部載る、検索カードに URI・作成者・日付が載る、検索ビューの
   フィルタ縦レールがサイドバーと二重に見える、詳細のパンくずの
   「knowledge」ラベルに意味がない、Review の Refresh ボタンが不要 — など。

## 2. move: ID の書き換えをサーバーの 1 トランザクションで行う

### 決定

`POST /api/v1/move {"from": <id>, "to": <id>}` を追加する。id は
階層パスなので `{id...}` ワイルドカードの後ろにサフィックスを置けず、
/usage・/revisions と同じ理由でトップレベルのパスを持つ。

move は「id を記録しているすべての場所の書き換え」であり、1 つの
トランザクションで行う:

- `knowledge` 本体の id(updated_at 更新)。
- `knowledge_revision` — **履歴はエントリに付いて移る**。move 自体も
  `change: "move"` のリビジョンとして追記する(スナップショットは新 id)。
- `attachment` / `knowledge_usage` / `knowledge_event` のキー。
- `knowledge_embedding` / `attachment_embedding`(存在すれば)。
  埋め込みテキストは id を含まないため、キーの付け替えだけで足り、
  再埋め込みは不要。
- **内向き参照の書き換え**: 生きているエントリのうち、リンク先が旧 id
  (素の形・`ochakai://` 形の両方)のもの、および `attrs.model` が旧 id の
  もの(0019 の compile モデル参照)。各書き換えは `change: "update"` の
  リビジョンとして記録する — 移動の副作用も監査証跡に残る。

添付のバイト列は動かない: blob は sha256 のコンテンツアドレス(0011)で、
エントリとの対応は `attachment` 行が持つ。

### 移動先の衝突

移動先に行が既にある場合は 409(ErrAlreadyExists)。**ソフト削除済みでも
拒否する** — その行はまだ主キーとリビジョン履歴を所有しており、上書きは
2 系統の履歴の合流になってしまう。復活は create の役目(既存規則)で、
move は常に空きアドレスへの移動である。

### スコープ

- REST + CLI(`ochakai move <id> <new-id>`)+ Web UI。**MCP には出さない**
  — 整理(キュレーション)は人間向けの操作で、0015 の面別ポリシーの
  とおりエージェントには search / get_context を残す。
- 本文内のエントリ相対な添付参照(`<末尾セグメント>/<name>`)は
  書き換えない。末尾セグメントが変わる移動では旧表記が残るが、表示側は
  ファイル名一致のフォールバックで解決する(0008)。エクスポート時の
  添付パスが新 id に変わるのはリネームの本義であり、リポジトリ内の
  ファイル改名と同じ扱いとする。

## 3. Web UI

### 3.1 移動の体験

- 詳細ビューに **Move** アクション: 現在のパスを種にした入力欄
  (ツリーが読み込み済みのディレクトリ + ルートを datalist で補完)
  + Move ボタン。
- ツリーの**ドラッグ & ドロップ**: エントリ行をディレクトリ行(または
  ツリー背景 = ルート)に落とすと、末尾セグメントを保って移動
  (confirm を挟む)。

### 3.2 ディレクトリ配下への新規作成

ディレクトリ行のホバーで ＋ を出し、`#/new/<prefix>/` に遷移。エディタの
ID 欄が prefix で埋まり、カーソルはその直後 — スラグだけ打てばよい。

### 3.3 表示の整理

- ボタン型リンク(`a.btn`)のホバー下線を抑制 — ＋ の下に下線が出ると
  プラスマイナスに見える。
- サイドバーの行は**タイトル + ステータスドット**に圧縮。id と正確な
  ステータスはツールチップと URL が持つ。verified は常態なのでドットも
  出さない(draft / deprecated / rejected のみ色ドット)。
- サイドバーはトップバーの ☰ で開閉でき、選択は localStorage に永続化
  (デスクトップのみ。モバイルは従来の disclosure)。
- 検索ビューのフィルタは縦レールをやめ、検索箱の下の**水平チップバー**に
  する — ツリーサイドバーとの二重レールを解消。狭い画面では従来どおり
  disclosure に畳む。
- 検索カードから URI・作成者・日付のメタ行を削除(タイトルのツールチップに
  URI を残す)。Review カードも URI を落とすが、作成者・経過日数・使用実績は
  キューの判断材料なので残す。
- 詳細のパンくずは `knowledge / <id>` をやめ、ルートへのリンクを兼ねる
  `/ <id>` にする。
- Review の Refresh ボタンを削除(stale トグルとナビゲーションで足りる)。

## 4. 互換性

- API は追加のみ(`POST /api/v1/move`、Revision.change に `"move"`)。
  既存クライアントに影響なし。マイグレーション不要。
- CLI はコマンド追加のみ。
