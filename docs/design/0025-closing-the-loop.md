# ochakai 設計ドキュメント 0025: 書き戻しループを締める — failed が再検証を呼ぶ

Status: Accepted(2026-07-22)
Date: 2026-07-22

## 1. 目的

ochakai の中核仮説は「エージェントが下書きを量産し、人間が検証する」
双方向ループである(0001)。このループは 3 つの部品が配線されて初めて
締まる:

1. **利用が測られている** — どのナレッジが実際に使われたか。
2. **検証が時効を持つ状態で、古くなると浮上する** — verified は
   スキーマ変更で腐る。
3. **再検証が証拠で優先され、人が回すワークフローの中で拾われる** —
   使われていて、かつ壊れている兆候のあるものから直す。

ochakai は 1 と 2 を既に持っている。利用テレメトリ(0001 §9.2、
`knowledge_usage`)、`report_outcome` による worked/failed の能動報告
(0001 §9.4)、検証 age フィード(`sort=verified_at`)と draft の
stale triage(`sort=usage`)である。

**欠けているのは 3 の一部である。** `failed`(結果が誤りだったという
能動報告)は集計され、Web UI の詳細画面のタイルには出るが、**「failed
報告が付いた verified エントリを再検証キューに浮上させる導線が無い」**。
検証 age フィードは `verified_at` の古さ(時間ベースの陳腐化)でしか
並ばず、*失敗報告*(証拠ベースの陳腐化)では並ばない。結果、あるエージェントが
golden query を実行して誤った数字を得て `report_outcome: failed` を
返しても、次のエージェントは同じ verified エントリを同じ信頼度で引いて
しまう。0001 §9.4 が自ら「書き戻しループの最後の一辺」と呼んだ穴である。

本ドキュメントはこの一辺を、**新しい型システムも LLM も実行も足さず、
既にある集計を surface するだけ**で締める。

## 2. 世界観: 時間ベースの陳腐化に、証拠ベースの陳腐化を足す

`sort=verified_at` は「長く検証されていない」を浮上させる —— *時間* が
陳腐化の代理指標である。だが「検証はしたが、実際に使ったら間違っていた」
エントリは、検証してから時間が経っていなくても最優先で直すべきだ。ここでは
*失敗報告* が陳腐化の直接の証拠になる。

両者は補完的である。時間ベースは「腐っているかもしれない」を、証拠ベースは
「腐っていると報告された」を拾う。本改訂は後者のフィードを 1 つ足す。

外部の裏付け(2026-07 の調査):
- Atlan の LLM ナレッジベース freshness スコアリングは、再検証を
  「実際に live クエリで surface しているもの(stale-retrieval rate)」で
  重み付けする —— corpus 全体ではなく、使われて壊れているものから直す。
- 記憶研究の "verify-then-freeze"(VerificAgent)は、無監査の自動記憶が
  内部で誤りを蓄積するのを人間の検証ゲートで止める。ochakai の検証ゲートは
  同型で、本改訂はそのゲートに「証拠ベースの入口」を足す。
- Meta の社内アナリティクスエージェント(4,500 レシピが 15 万回利用)や
  dbt の analytics engineer が示すのは、生きたループは「利用が保守価値を
  集中させ、curation を人が回すワークフローの門にする」ことで締まる、という
  一点である。ochakai の canary 運用(client 実行)がその門であり、本改訂は
  そこに「failed の付いたものも再検証対象」という材料を供給する。

## 3. 決定

### 3.1 `sort=failed` 再検証フィードを足す

`GET /api/v1/knowledge`、MCP `search_knowledge`、CLI `search` の `sort` に
`failed` を追加する。挙動は既存の 2 フィード(`verified_at` / `usage`)と
対称:

- 検索ではなく列挙である。クエリ `q` とは排他(併用は 400 / エラー)。
- 各ヒットは `usage`(worked / failed を含む集計)を同梱する。`usage`
  フィードと同じ射影であり、レビュアーが証拠をインラインで見られる。
- score は 0(列挙であって検索ではない)。
- フィルタ(type / status / tag)は従来どおり合成する。`status=verified`
  を渡せば「検証済みで、かつ失敗報告のあるもの」に絞れる。

### 3.2 並び順と対象

**対象は `failed > 0` のエントリのみ。** 失敗報告がゼロなら、フィードは
空になる —— 「誤りと報告されたものは無い」を正直に返す。新規インスタンスや
健全な状態では空であり、これは正しい挙動である。

**並び順:** `failed DESC, worked ASC, verified_at ASC NULLS LAST, id`。

- `failed DESC` — 最も多く誤りと報告されたものが先頭。
- `worked ASC` — 同じ失敗数なら、裏付けの worked が少ないもの(信号対雑音が
  最悪)を上に。
- `verified_at ASC NULLS LAST` — さらに同じなら、検証が古いもの(定義が
  ドリフトした可能性が高い)を上に。**未検証(draft)は最後**: 認証されて
  いない下書きの失敗より、認証済みエントリの失敗の方が緊急である。
- `id` — 安定なタイブレーク。

### 3.3 集計を使う(生イベントの時刻には依存しない)

並びは `knowledge_usage` の集計(count と last_at)だけで計算する。
`knowledge_event` の per-event 時刻(180 日で刈り取られる、0001 §9.2)には
依存しない。「verified_at より後に付いた失敗か」を厳密に判定する案(§4)は
採らず、失敗**回数**を証拠とする。集計は無期限に残るので、フィードは古い
エントリでも正しく動く。

### 3.4 DB 変更なし

`failed` イベントは 0005(outcomes マイグレーション)で既に
`knowledge_usage` に集計されている。本改訂はクエリの追加のみで、スキーマ
移行を持たない。

### 3.5 各サーフェスの更新

- **store**: `ListByFailed`(`ListByUsage` と同じ LATERAL 射影、WHERE に
  `u.failed > 0`、§3.2 の ORDER BY)。
- **service**: `ListByFailed`(limit 既定 100・上限 1000。列挙なので利用は
  記録しない —— triage がランキング対象の信号を膨らませないため)。
- **restapi / mcpserver / CLI**: `sort` の妥当値に `failed` を足し、
  ツール説明・ヘルプ・補完(zsh / bash / fish)・openapi の enum を更新。
- **Web UI**: Explore に「needs review」トグルを足す(検証 age トグルと
  対称・排他)。オンで `sort=failed`。フィードのバナーで用途を説明する。

## 4. やらないこと

- **自動 demote / status ゲート。** failed が閾値を超えても verified を
  自動で draft に落とさない。0002「信頼は provenance から判断し、遷移を
  権限で縛らない」を維持する。フィードは*浮上させる*だけで、直すか否かは
  人間/エージェントの判断に委ねる。
- **「verified_at より後の失敗」の厳密判定。** `knowledge_event` の
  per-event 時刻を verified_at と突き合わせれば「検証後に壊れた」を厳密に
  出せるが、生イベントは 180 日で刈られ、集計だけでは再現できない。失敗
  回数で足りるので、この精緻化は見送る(必要が定量的に見えたら再検討)。
- **ランキングへの反映。** 通常検索の並びに failed を混ぜない(0001 §9.4:
  failed が積んだエントリはフィードとカナリアで人/エージェントが拾う)。
- **LLM・実行・型付き関係・オントロジー層。** 本改訂は既存集計の surface で
  あり、ochakai の中核制約(LLM ゼロ・非実行)も、リンクを型無しに保つ
  方針(0024)も、オントロジー層を採らない方針(0001 §4・§9.5-4)も変えない。
- **カードへの失敗数のインライン表示。** `usage` フィードのカードが利用数を
  出していないのと同じく、`failed` フィードのカードも既存の hitCard のまま
  にする。証拠は詳細画面の worked/failed タイルで見える。統一を優先する。

## 5. 壊れるもの

追加のみなので破壊は最小:

- `sort` の enum が 1 値増える(`verified_at` / `usage` / `failed`)。旧
  クライアントには影響しない。未知の `sort` は従来どおりエラー。
- openapi・ツール説明・CLI ヘルプ・シェル補完の文言が増える。
- 既存の 2 フィードの挙動・ワイヤ形式は不変。
